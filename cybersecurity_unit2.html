Topic 17

Computer Fundamentals

🖥️

Definition

Definition
A computer is an electronic device that accepts input data, processes it according to a set of instructions (programs), stores the results, and produces output. The basic model is: Input → Process → Output → Storage.

Every digital device — from a smartphone to a supercomputer — follows this same fundamental model. Understanding it is the foundation of all cybersecurity thinking because each stage is an attack surface.

IPO Cycle (Input → Process → Output)

📥 INPUT
Keyboard, Mouse, Mic
→
⚙️ PROCESS
CPU Executes Code
→
📤 OUTPUT
Monitor, Speaker, Printer
⇅
💾 STORAGE
HDD, SSD, RAM

Key Components

🧠
CPU
Central Processing Unit — the "brain". Executes instructions. Speed measured in GHz (billion cycles/sec). Has cores: dual, quad, octa.
🗃️
RAM
Random Access Memory — temporary, fast working memory. Lost on power off. Measured in GB. More RAM = more programs run simultaneously.
💽
Storage (HDD/SSD)
Permanent data storage. HDD = magnetic spinning disk, slower. SSD = flash memory, faster, more expensive, no moving parts.
🎛️
Motherboard
The main circuit board connecting all components. Contains BIOS/UEFI firmware. First thing to start when PC powers on.
🖥️
GPU
Graphics Processing Unit — renders images/video. Also used for ML and password cracking (parallel processing power).
🔌
PSU
Power Supply Unit — converts AC mains power to DC power for components. Failure = total system failure.

Number Systems — Binary & Hex

Computers operate in binary (base-2). All data is ultimately 0s and 1s. Hackers also use hexadecimal (base-16) to represent binary data compactly.

NUMBER SYSTEMS

🔬 Exercise
  1. Convert the decimal number 172 to binary and hexadecimal.
  2. A website's IP is 192.168.0.1 — convert each octet to binary. How many total bits in an IPv4 address?

📌 Conclusion

Every component in a computer is both a tool and a potential vulnerability. Understanding the hardware fundamentals lets you think like both a builder and an attacker — essential for any cybersecurity professional.

Topic 18

Hardware vs Software

⚙️

Definitions

🔧 Hardware

Physical, tangible components you can touch.

  • CPU, RAM, HDD, SSD
  • Monitor, keyboard, mouse
  • Network cards, routers
  • Firmware (software burned into hardware)
  • Cannot be easily modified/patched

💾 Software

Intangible programs and data; instructions for hardware.

  • Operating Systems (Windows, Linux)
  • Applications (Chrome, Word)
  • Drivers (translate OS ↔ hardware)
  • Middleware, APIs, libraries
  • Easily updated/patched

Software Categories

CategoryDescriptionExamplesSecurity Note
System SoftwareManages hardware resourcesWindows, Linux, macOS, BIOSMost critical to patch — runs with highest privilege
Application SoftwareUser-facing programsChrome, Notepad, PhotoshopLargest attack surface (most bugs)
Utility SoftwareSystem maintenance toolsAntivirus, disk cleaner, backupAntivirus itself can have vulnerabilities
Programming SoftwareTools to write softwareVisual Studio, Python, GCCSupply chain attacks target IDEs/compilers
FirmwareLow-level software in hardwareUEFI, router firmware, SSD controllerHard to patch; rootkits live here
MalwareMalicious softwareVirus, worm, ransomware, trojanExploits all above categories

Software Vulnerability Lifecycle

Developer Writes Code
Bug / Vulnerability Introduced
Researcher Discovers Vulnerability
Responsible Disclosure?
Vendor Creates Patch
Patch Released — Users Must Apply It!
Vulnerability Mitigated (if patched)
⚠️ Zero-Day Window: The time between vulnerability discovery and patch release. During this window, no defence exists except workarounds. WannaCry exploited an unpatched Windows vulnerability (EternalBlue) — Microsoft had released a patch 2 months earlier, but many users hadn't applied it.

Advantages & Disadvantages

✅ Software Advantages

  • Easily updated/patched remotely
  • Can be replicated infinitely at zero cost
  • Open source allows community security review
  • Virtualisation runs multiple systems on one hardware

⚠️ Software Disadvantages

  • Vulnerable to bugs, exploits, malware
  • Complexity increases attack surface
  • Pirated software often contains backdoors
  • Dependency on unvetted libraries (supply chain risk)
🔬 Exercise
  1. Classify each item: browser plugin, USB drive, network switch, operating system kernel, UEFI firmware — is it hardware, software, or firmware?
  2. A ransomware attack encrypts files on your computer. Is the ransomware hardware or software? What part of the IPO model does it target?
📌 Conclusion

Hardware provides the physical foundation; software provides the intelligence. Most cyber attacks target software because it can be manipulated remotely without physical access. Keeping both hardware firmware and software patched is your primary defence.

Topic 19

Operating System Basics

🖱️

Definition

Definition
An Operating System (OS) is the master control software that manages all hardware resources, provides services to applications, and acts as an interface between the user and the hardware. Without an OS, you cannot run any programs.

Core OS Functions

⚙️
Process Management
Creates, schedules, and terminates processes. Ensures CPU time is shared fairly (multitasking).
🗄️
Memory Management
Allocates RAM to processes. Uses virtual memory (swap) when RAM is full. Prevents processes accessing each other's memory.
📁
File System Mgmt
Organises files in hierarchical structure. Controls read/write/execute permissions on files and directories.
🔌
Device Management
Communicates with hardware via device drivers. I/O scheduling for disks, network cards, USB devices.
👤
User Management
Manages user accounts, passwords, access rights. Enforces privilege levels (admin vs standard user).
🔒
Security & Access
Enforces file permissions, user authentication, process isolation. First line of defence on any system.

OS Architecture: Kernel vs User Space

# User apps run in Ring 3 (lowest privilege)

File Permission System (Linux Example)

LINUX PERMISSIONS
# Command: ls -la
-rwxr-xr--  2  alice  staff  4096  Mar 15  password_checker.py

Position:  -  rwx  r-x  r--
           │   │    │    └── Others: read only
           │   │    └─────── Group:  read + execute
           │   └──────────── Owner:  read + write + execute
           └──────────────── File type (- = file, d = directory)

# Numeric representation (chmod):
r=4, w=2, x=1
chmod 755 file  →  rwx r-x r-x  (owner=7, group=5, others=5)
chmod 600 file  →  rw- --- ---  (owner only — use for private keys!)

# BAD PRACTICE (never do this):
chmod 777 /var/www/html  →  EVERYONE can read, write, execute!
🔬 Exercise
  1. What chmod number would give owner read+write, group read only, others no permissions?
  2. An attacker gains kernel-level access. What can they do that a normal user-space attacker cannot?
  3. List three ways an OS protects one process from interfering with another.
📌 Conclusion

The OS is the gatekeeper of all system resources. Compromising the OS kernel grants an attacker complete control over the machine. This is why privilege escalation attacks — gaining admin/root access — are among the most dangerous exploits.

Windows vs Linux vs macOS Security

🪟 Windows

  • ~75% desktop market share = #1 malware target
  • Active Directory for enterprise identity
  • Many legacy applications = old vulnerabilities
  • Windows Defender built-in AV
  • SMB protocol historically exploited (WannaCry)

🐧 Linux

  • Open source — community audited
  • Dominates servers (90%+ of web servers)
  • Fine-grained permissions (everything is a file)
  • Fewer consumer malware, but targeted by APTs
  • Kali Linux = standard pen-testing OS

🍎 macOS

  • Unix-based — inherits Linux security model
  • Growing malware target as market share rises
  • Gatekeeper prevents unsigned app installation
  • SIP (System Integrity Protection) blocks rootkits

📱 Android vs iOS

  • Android: open, more malware risk, APK sideloading
  • iOS: closed, App Store only, fewer vulnerabilities
  • Both: app sandboxing limits damage
  • Both: full disk encryption enabled by default
🔬 Exercise
  1. A hospital runs Windows XP on medical devices because upgrading is expensive. What security risks does this create? What can they do to reduce risk?
  2. Why would a penetration tester choose Kali Linux over Windows for their work machine?
📌 Conclusion

No OS is 100% secure. The "most secure" OS depends on the use case, how it's configured, and how actively it's patched. The most dangerous OS is always the one that's out of date and unpatched.

File Systems

📁

Definition

Definition
A file system is the method and data structure an OS uses to organise, store, retrieve, and manage files on a storage device. It defines how data is named, stored, and accessed — and who can do what to it.

Common File Systems

File SystemOSMax File SizeFeaturesSecurity Notes

FAT32USB drives4 GBUniversal compatibility, simpleNo permissions — anyone can read/write everything

ext4Linux16 TBJournaling, permissions, matureStrong Unix permissions; supports SELinux labels
APFSmacOS/iOS8 EBSnapshots, encryption, cloningFull encryption at rest, copy-on-write

Directory Structure

FILE SYSTEM PATHS
# Windows (NTFS) — backslash separator
C:\                    ← Root (C drive)
C:\Windows\System32\   ← OS system files (protect this!)
C:\Users\Alice\        ← User home directory
C:\Program Files\      ← Installed applications

# Linux (ext4) — forward slash separator
/                  ← Root (everything starts here)
/etc/              ← System config files (passwd, shadow, hosts)
/home/alice/       ← User home directory
/var/log/          ← System logs (security logs live here!)
/tmp/              ← Temporary files (often used by malware!)
/etc/shadow        ← Hashed passwords — NEVER world-readable!

# Attacker's interest:
C:\Windows\SAM     ← Windows password hashes
/etc/shadow        ← Linux password hashes
~/.ssh/id_rsa      ← Private SSH keys

NTFS Alternate Data Streams (ADS) — Steganography Attack

🚨
NTFS ADS Attack: Windows NTFS allows files to have "hidden streams" attached. Malware can hide itself inside a legitimate file's ADS. E.g., notepad.exe:malware.exe — antivirus may miss this! Detection: use dir /r or tools like Streams.exe.
🔬 Exercise
  1. Why is FAT32 dangerous for a USB drive containing sensitive work documents?
  2. An attacker gains read access to /etc/shadow. What can they do with it? What attack follows?
  3. Which file system would you choose for a forensics evidence server and why?
📌 Conclusion

File systems are not just about organisation — they are critical security boundaries. Misconfigurations (world-readable sensitive files, no encryption on USB drives) are among the easiest vulnerabilities for attackers to exploit.

Topic 22

Process & Memory Management

🧠

Process Management

Definition
A process is a program in execution. It has its own memory space, CPU time, and resources. The OS manages the lifecycle of processes from creation to termination.
Process StateDescriptionSecurity Implication
NewProcess being createdProcess injection at creation time
ReadyWaiting for CPU timeScheduler manipulation
RunningCurrently executing on CPUMemory scraping (POS malware)
WaitingBlocked waiting for I/ORace conditions (TOCTOU attacks)
TerminatedFinished executionMemory not cleared = data leakage

Memory Layout & Buffer Overflow Attack

MEMORY LAYOUT & BUFFER OVERFLOW
# Program memory layout (stack grows down):
┌─────────────────────────────┐  HIGH ADDRESS
│  Command Line Args / Env Vars │
├─────────────────────────────┤
│         Stack               │  ← grows downward ↓
│   (local variables, return  │
│    addresses, stack frames) │  ← BUFFER OVERFLOW TARGET
├─────────────────────────────┤
│           ↕ gap             │
├─────────────────────────────┤
│          Heap               │  ← grows upward ↑
│   (dynamically allocated)   │
├─────────────────────────────┤
│   BSS Segment (uninitialized)│
├─────────────────────────────┤
│  Data Segment (global vars) │
├─────────────────────────────┤
│    Text Segment (code)       │
└─────────────────────────────┘  LOW ADDRESS

# Buffer Overflow Example (C — vulnerable code):
char buffer[8];  // only 8 bytes allocated
strcpy(buffer, user_input);  // NO bounds check!

# If attacker sends 20 bytes:
AAAAAAAAAAAAAAAAAAAA  ← overflow overwrites return address!
buffer  ↑ OVERWRITES saved RIP → attacker controls code flow

# Defences:
ASLR  = randomize memory addresses (makes overflow harder)
DEP   = Data Execution Prevention (stack not executable)
Stack Canaries = detect overflow before return
Safe functions: strncpy(), snprintf() instead of strcpy(), sprintf()

Virtual Memory & Paging

💡
Virtual Memory: The OS gives each process the illusion it has its own private memory space. Physical RAM is divided into 4KB "pages" mapped to virtual addresses. When RAM is full, infrequently used pages are moved to disk (swap/pagefile). Security benefit: process isolation — Process A cannot read Process B's memory directly.
🔬 Exercise
  1. What is a buffer overflow attack? Name one famous vulnerability caused by it.
  2. Why does virtual memory improve security even if it doesn't add encryption?
  3. An attacker writes a program that fills a 256-byte input field with 512 'A' characters. What vulnerability are they testing for?
📌 Conclusion

Memory is where all running code lives — it's the battlefield of exploitation. Buffer overflows were responsible for decades of critical vulnerabilities. Modern defences (ASLR, DEP, canaries) have made exploitation harder but not impossible.

Topic 23

Internet Basics

🌐

What is the Internet?

Definition
The Internet is a global system of interconnected computer networks that communicate using a standardised protocol suite (TCP/IP). It is not owned by any single entity — it is a network of networks connecting billions of devices worldwide.

🌐 Internet

  • Global public network
  • Anyone can connect (with ISP)
  • Carries all types of traffic (email, web, video)
  • Not owned by anyone
  • Routes data around failures (designed for nuclear war)

🕸️ Web (WWW)

  • Service running on the Internet
  • HTTP/HTTPS websites and pages
  • Invented by Tim Berners-Lee (1989, CERN)
  • Requires a browser to navigate
  • Web ≠ Internet (email ≠ web, but uses internet)

Topic 24

How the Internet Works

📡

TCP/IP Protocol Suite

All internet communication uses TCP/IP — a set of rules (protocols) governing how data is packaged, addressed, transmitted, routed, and received.

7ApplicationHTTP, HTTPS, FTP, SMTP, DNS, SSH — user-facing protocols
6PresentationEncryption/decryption (SSL/TLS), data formatting (JSON, XML)
5SessionManages connections and sessions between apps
4TransportTCP (reliable, ordered) | UDP (fast, no guarantee) — ports here
3NetworkIP addressing & routing — packets, routers, ICMP
2Data LinkMAC addresses, Ethernet frames, switches
1PhysicalCables, radio waves, fibre optic — actual bits as signals
🧠 Memory trick: "All People Seem To Need Data Processing" (Application, Presentation, Session, Transport, Network, Data Link, Physical) — or in reverse "Please Do Not Throw Sausage Pizza Away"

What Happens When You Visit a Website

You type: https://google.com
1️⃣ DNS Lookup: google.com → 142.250.80.46
2️⃣ TCP 3-Way Handshake with 142.250.80.46:443
3️⃣ TLS Handshake — certificates verified, session key exchanged
4️⃣ HTTP GET / request sent (encrypted)
5️⃣ Server responds with HTTP 200 + HTML
6️⃣ Browser renders HTML, loads CSS/JS/Images
Page displayed to user (~50–200ms)

TCP vs UDP

FeatureTCPUDP
ConnectionConnection-oriented (3-way handshake)Connectionless
ReliabilityGuaranteed delivery, ordered packetsBest-effort, packets can be lost
SpeedSlower (overhead)Faster (no overhead)
Use casesHTTP, HTTPS, SSH, email, file transferDNS, VoIP, gaming, video streaming
Attack riskSYN flood (exhaust half-open connections)UDP flood (DDoS amplification)
🔬 Exercise
  1. At which OSI layer does a firewall typically operate? What about a switch? A proxy?
  2. A SYN flood attack sends millions of SYN packets but never completes the handshake. At which OSI layer does this attack occur and why is it effective?
📌 Conclusion

The OSI model is the cybersecurity professional's map. Every attack can be located on this model, and defences are designed layer by layer. Know where an attack lives in the OSI model and you'll know what tools to use against it.

Client–Server Model

🖧

Definition

Definition
The client–server model is a distributed application structure where the client (requester) sends requests to a server (provider), which processes them and returns a response. The internet is built almost entirely on this model.

🔬 Exercise
  1. When you log into your bank's website, which component is the client and which is the server? What port does the connection use?

📌 Conclusion

Attackers target every layer of the client-server relationship. Defence must cover the client (browser security), the channel (encryption/HTTPS), and the server (patching, WAF, input validation). Security is only as strong as the weakest link.

How a Browser Works

A web browser is a client application that retrieves, parses, and renders web content. Understanding how browsers work is fundamental to web security.

Browser Security Features

🏖️
Sandbox
Each browser tab runs in an isolated sandbox. JavaScript cannot access your filesystem, OS, or other tabs' data.
🔒
Same-Origin Policy
JavaScript on evil.com cannot read data from bank.com. Prevents cross-site data theft. CORS relaxes this when needed.
🚨
Safe Browsing
Google/Firefox maintain blocklists of phishing/malware sites. Browser warns before visiting dangerous URLs.
📜
CSP Headers
Content Security Policy: server tells browser which sources are allowed to load scripts, images, etc. Prevents XSS.
🍪
Cookie Controls
Browsers enforce SameSite, Secure, HttpOnly cookie flags to prevent theft and CSRF attacks.
🔐
HTTPS Enforcement
HSTS (HTTP Strict Transport Security) forces browsers to always use HTTPS for a site, preventing downgrade attacks.

Browser Fingerprinting — Privacy Attack

👁️
Browser Fingerprinting: Websites can identify you without cookies by combining: screen resolution, installed fonts, browser plugins, timezone, language, canvas rendering, WebGL data. Panopticlick found 83% of browsers had a unique fingerprint. Used by ad trackers and criminals to track users across sites even in "incognito" mode.

Search Engines & OSINT

📌 Conclusion

Browsers are both gateways to the internet and significant attack surfaces. Modern browsers include robust security features — but attackers exploit misconfigurations (bad CSP, missing HttpOnly flags) and browser bugs. Keep your browser updated; it's patched constantly.

Topic 28

HTTPS & SSL/TLS Basics

🔒

Definition

Definition
HTTPS (HTTP Secure) = HTTP + TLS (Transport Layer Security). TLS is the cryptographic protocol that provides three security guarantees: Confidentiality (encryption), Integrity (tamper detection), and Authentication (certificate verification). The old name "SSL" (Secure Sockets Layer) is deprecated but still used colloquially.

TLS Handshake — Step by Step

Client sends: ClientHello (TLS version, cipher suites, random number)
Server sends: ServerHello (chosen cipher suite) + Certificate
Client verifies certificate with Certificate Authority (CA)
Key exchange: both derive shared session key (using ECDHE)
Both send Finished message — handshake complete
🔐 All data now encrypted with AES-256 (symmetric)

Digital Certificates & PKI

SSL CERTIFICATE ANATOMY
# What's in an SSL certificate (X.509 format):

Issuer:         CN=GTS CA 1C3, O=Google Trust Services LLC
Valid From:     2024-01-15
Valid To:       2024-04-08

# Certificate Trust Chain:
Root CA (pre-installed in OS/browser)
  └─▶ Intermediate CA
        └─▶ Website Certificate (google.com)

# Certificate Attacks:
- Fake/self-signed cert → Browser shows warning (heed it!)
- Expired cert        → Connection should be refused
- Certificate pinning → App checks specific cert fingerprint
- CA compromise      → DigiNotar (2011) — Iranian gov issued fake Google certs
✅
Free Certificates: Let's Encrypt (letsencrypt.org) provides free, automated TLS certificates, removing the excuse of "it's too expensive to use HTTPS." In 2024, ~82% of all web traffic is encrypted with HTTPS.

📌 Conclusion

HTTPS is non-negotiable for any site handling user data. A padlock icon does NOT mean a site is safe — it only means the connection is encrypted. The certificate must be from a trusted CA, the site content must be legitimate, and TLS 1.2+ must be used.

Topic 29

Cookies & Sessions

🍪

Definitions

🍪 Cookies

Small text files stored by browser, sent with every request to the domain that set them.

  • Set by server via Set-Cookie header
  • Sent by browser via Cookie header
  • Can persist after browser closes (persistent)
  • Used for sessions, preferences, tracking
  • Max ~4KB per cookie

🎫 Sessions

Server-side mechanism to remember state across stateless HTTP requests.

  • Server stores session data (username, cart, role)
  • Client receives only a session ID (random token)
  • Ends when browser closes or times out
  • Session ID typically stored in a cookie
  • Invalidated on logout

Cookie Security Flags

SECURE COOKIE HEADER
# Insecure cookie (vulnerable):
Set-Cookie: session=abc123

# Secure cookie (hardened):
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=3600

Flag explanation:
HttpOnly    → JavaScript CANNOT read this cookie (prevents XSS cookie theft)
Secure      → Cookie ONLY sent over HTTPS (prevents MitM theft)
SameSite=Strict → Cookie NOT sent in cross-site requests (prevents CSRF)
Max-Age=3600    → Cookie expires after 1 hour
Path=/          → Cookie valid for entire site

# Attack: Session Hijacking
1. Attacker steals session cookie (XSS, sniffing, MitM)
2. Attacker sends request with stolen cookie
3. Server thinks attacker IS the victim user
4. Attacker has full account access!

# Defense: Bind session to IP + User-Agent + use HttpOnly + Secure

CSRF — Cross-Site Request Forgery

⚠️
CSRF Attack: Victim is logged into bank.com. Attacker tricks victim into clicking a link to evil.com, which secretly submits a form to bank.com/transfer?amount=10000&to=attacker. Browser automatically sends the session cookie! Defence: CSRF tokens (secret value in form that attacker can't guess) + SameSite cookies.
🔬 Exercise
  1. A cookie is set without the HttpOnly flag. Write the JavaScript an attacker could use to steal it via XSS.
  2. Why does SameSite=Strict prevent CSRF attacks? What is the trade-off?
  3. What is the difference between a session cookie and a persistent cookie?
📌 Conclusion

Cookies power web authentication — and are among the most attacked web assets. Always set HttpOnly, Secure, and SameSite flags. Use short session timeouts. Invalidate sessions on logout. Never put sensitive data directly in a cookie.

Topic 30

DNS Basics

📖

Definition

Definition
DNS (Domain Name System) is the internet's "phone book" — it translates human-readable domain names (like google.com) into IP addresses (like 142.250.80.46) that computers use to route traffic. Without DNS, you'd have to remember the IP addresses of every website.

DNS Resolution Process

User types: www.example.com
1️⃣ Check local cache + /etc/hosts file
2️⃣ Query Recursive Resolver (your ISP's DNS, e.g. 8.8.8.8)
3️⃣ Resolver queries Root Server (13 root servers globally)
4️⃣ Root says: "Ask .com TLD nameserver"
5️⃣ TLD (.com) says: "Ask example.com's authoritative NS"
6️⃣ Authoritative NS returns IP: 93.184.216.34
Browser connects to 93.184.216.34:443

DNS Record Types

RecordPurposeExampleSecurity Note
ADomain → IPv4 addressexample.com → 93.184.216.34Target of DNS spoofing
AAAADomain → IPv6 addressexample.com → 2606:2800:220:1::/32IPv6 attack surface
CNAMEAlias → another domainwww → example.comSubdomain takeover risk

PTRReverse DNS (IP → domain)34.216.184.93.in-addr.arpaUsed in spam filtering

DNS Attacks

DNS ATTACK TYPES
1. DNS Cache Poisoning (Kaminsky Attack 2008)
   Attacker injects fake DNS record into resolver's cache
   Victim queries resolver → gets FAKE IP → visits attacker's server
   Defence: DNSSEC (digitally signed DNS records), DNS-over-HTTPS (DoH)

2. DNS Hijacking
   Malware changes your router's DNS server to attacker-controlled server
   All DNS lookups return attacker's IPs
   Defence: Router strong password, check DNS settings regularly

3. DNS Amplification (DDoS)
   Attacker sends small DNS queries with spoofed source IP (victim's IP)
   DNS servers send 50-100x larger responses to victim
   3000 Gbps DDoS attacks have been launched this way
   Defence: Rate limiting, response rate limiting (RRL), BCP38

4. DNS Tunnelling
   Malware encodes C2 data in DNS queries (base64 encoded)
   Bypasses firewall — DNS port 53 is almost always open!
   Detection: Unusually long DNS hostnames, high DNS query volume

Tools for DNS Recon:
nslookup example.com    → Basic DNS query
dig example.com ANY      → All DNS records
dig -x 93.184.216.34     → Reverse DNS lookup
fierce -dns example.com  → Subdomain enumeration
🔬 Exercise
  1. What is DNS cache poisoning and how does DNSSEC prevent it?
  2. An attacker buys the domain "paypa1.com" (with the number 1 instead of the letter l). What attack technique is this? How does DNS make this possible?
  3. Why is DNS tunnelling hard to block with a standard firewall?
📌 Conclusion

DNS is critical infrastructure — if an attacker controls your DNS, they control where all your traffic goes. Enable DNSSEC, use trusted DNS servers (1.1.1.1, 8.8.8.8), use DNS-over-HTTPS for privacy, and monitor for unusual DNS activity.

Topic 31

Internet Security Basics

🛡️

Core Internet Security Controls

🔥
Firewall
Filters traffic based on rules (IP, port, protocol). Stateful firewalls track connection state. NGFW does deep packet inspection. Block inbound by default; allow outbound selectively.
🕵️
IDS / IPS
IDS = detects attacks, alerts only. IPS = detects AND blocks. Signature-based (known attacks) vs anomaly-based (unusual behaviour). Snort, Suricata, Zeek.
🌐
VPN
Virtual Private Network — encrypts all traffic between device and VPN server. Hides real IP, protects on public WiFi. Not a silver bullet — VPN provider can log traffic.
🔍
Web Proxy
Intermediary between client and internet. Caches content, filters malicious sites, inspects HTTPS (MITM proxy). Used by enterprises to control web access.
🚫
Web Filter / DNS Filter
Blocks access to malicious/inappropriate sites by category. Cisco Umbrella, OpenDNS — blocks at DNS level before connection made.
📧
Email Security
SPF (who can send email for domain) + DKIM (email signing) + DMARC (policy on failures) = email authentication trinity preventing spoofing.

AttackHow it WorksLayerDefence
PhishingFake emails/sites steal credentialsHuman / AppEmail filtering, security training, MFA

Zero Trust Network Access (ZTNA)

🔐
Zero Trust: Traditional security assumed everything inside the network perimeter was safe. Zero Trust assumes the network is always hostile — every access request must be authenticated and authorised, regardless of source. Principles: verify identity continuously, use least privilege, micro-segment networks, assume breach. Companies like Google (BeyondCorp), Microsoft, and Cloudflare have migrated to ZTNA.
🔬 Exercise
  1. Explain the difference between an IDS and an IPS. Which generates more alerts that need investigation?
  2. A company's server is hit by a DDoS attack generating 500 Gbps of traffic. What solutions can mitigate this?
  3. Draw a simple network diagram showing where a firewall, IDS/IPS, and DMZ would be placed in a corporate network.
📌 Conclusion

Internet security requires a defence-in-depth approach — no single tool is enough. Layer firewalls, IDS/IPS, email security, DNS filtering, VPN, and user training. Assume every layer will eventually be breached and ensure you can detect and respond quickly.

Quick Reference Glossary

CPUBrain of computer. Executes instructions. Speed in GHz, measured in cores.
RAMTemporary fast memory. Lost on power off. More RAM = more apps can run.
OS (Kernel)Manages hardware. Kernel runs Ring 0 (highest privilege). Compromising kernel = full control.
Buffer OverflowWriting beyond allocated buffer overwrites memory. Can redirect code execution to attacker's payload.
File SystemOrganises storage. NTFS (Windows), ext4 (Linux), APFS (Mac). Controls permissions.
TCP vs UDPTCP = reliable, ordered, slow. UDP = fast, unreliable. HTTP uses TCP; DNS/VoIP use UDP.
OSI Model7 layers. Each layer = different attacks. Firewall = layer 3/4. XSS = layer 7 (application).
DNSTranslates domain names to IPs. Port 53. Vulnerable to cache poisoning, hijacking, tunnelling.
HTTPWeb protocol. Plaintext (dangerous). Methods: GET, POST, PUT, DELETE. Stateless (needs cookies).
HTTPS / TLSHTTP + encryption. TLS 1.3 is current standard. Certificate verifies server identity.
Cookie FlagsHttpOnly (no JS access), Secure (HTTPS only), SameSite (no CSRF). All three = hardened cookie.
Session HijackingSteal session cookie → impersonate user. Prevented by HttpOnly + Secure + short timeouts.
XSSInject JavaScript into page. Runs in victim's browser. Steals cookies, redirects, keylogging.
CSRFTrick authenticated user's browser to make request. Prevented by CSRF tokens + SameSite=Strict.
DNS PoisoningFake DNS records redirect traffic to attacker. DNSSEC/DoH prevents this.
FirewallFilters traffic by rules. Stateful tracks connection. NGFW does deep inspection.
VPNEncrypts tunnel to VPN server. Hides real IP. Protects on public WiFi. Not anonymous.
Zero TrustNever trust, always verify. Every access request authenticated regardless of network location.
ASLR / DEPAddress Space Layout Randomisation + Data Execution Prevention — OS defences against buffer overflows.
SPF/DKIM/DMARCEmail authentication. SPF = allowed senders. DKIM = signing. DMARC = policy. Together prevent spoofing.

📌 Unit 2 Final Takeaway

Computers and the internet were built for function, not security. Every layer — from the hardware to the browser — has known vulnerabilities.
