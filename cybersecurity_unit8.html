Topic 129

Ethical Hacking Introduction

🎩
⚡ One-Line Summary
Ethical hacking is the authorised, legal practice of attacking systems to find vulnerabilities before malicious hackers do — using the same tools and techniques but with written permission and a defined scope.

Definition

Definition
An ethical hacker (penetration tester / white-hat hacker) is a cybersecurity professional who uses hacking techniques with explicit written permission to test and improve the security of systems. They think like attackers but act as defenders. The entire practice is governed by legal agreements, defined scope, and professional ethics.

Why Ethical Hacking Exists

🔍
Find Before Attackers
Discover vulnerabilities in controlled conditions before real attackers exploit them in the wild.
✅
Verify Defences
Test whether security controls actually work under real attack conditions. Theory vs reality.
📋
Compliance
PCI-DSS, ISO 27001, GDPR, SOC 2 all require regular penetration testing as a compliance requirement.
📊
Risk Assessment
Quantify real risk: what can an attacker actually access? What is the business impact?
🧠
Security Awareness
Social engineering tests reveal gaps in human defences — training programs follow findings.
🛡️
Improve Architecture
Finding that segmentation failed, or logging missed attacks, drives architectural improvements.

The Three Core Requirements

ETHICAL HACKING REQUIREMENTS
1. AUTHORISATION (Most Important)
   Written permission from system owner BEFORE any testing
   Scope document: what's allowed, what's off-limits, when, from where
   Without this: CRIMINAL OFFENCE regardless of intent

2. DEFINED SCOPE
   Specific targets: which IPs, domains, applications
   Allowed techniques: black box / grey box / white box
   Time window: testing hours (avoid peak business hours)
   Out of scope: third-party systems, production databases, DoS testing

3. NON-DISCLOSURE
   NDA (Non-Disclosure Agreement) protects both parties
   Findings are confidential — not shared publicly
   Responsible disclosure if finding is serious

# Ethical hacker's professional obligations:
→ Stay within scope at ALL times
→ Do not access/modify data unless necessary to prove vulnerability
→ Report EVERYTHING found — including what works well
→ Preserve evidence in a forensically sound manner
→ Stop immediately if you accidentally find evidence of a crime
→ Follow Rules of Engagement (RoE) document

Testing Methodologies

TypeKnowledge LevelSimulatesBest For
Black BoxZero prior info (like real attacker)External attacker, zero inside knowledgeTesting external-facing defences, realistic attack simulation
Grey BoxPartial info (some credentials, network diagrams)Insider threat, or attacker with some reconEfficient, realistic — most common in practice
White BoxFull info (source code, all credentials, architecture)Privileged insider; code auditMaximum coverage, finding deepest vulnerabilities, compliance

🔬 Exercise
  1. Your friend asks you to "test" their school's network to "help them improve security." They have not gotten written permission from the school. Is this ethical hacking? What are the legal risks?
  2. A penetration tester is testing a company's web application and accidentally discovers the company's private encryption keys. What should they do?
📌 Conclusion

Ethical hacking exists because the best way to find weaknesses is to try to exploit them. The difference between a criminal hacker and an ethical hacker is: written authorisation, defined scope, and professional conduct. Everything else — the tools, techniques, and mindset — is identical.

Topic 131

Hacking Methodology

🗺️
⚡ One-Line Summary
Hacking follows a structured 5-phase methodology: Reconnaissance → Scanning → Enumeration → Exploitation → Post-Exploitation — ethical hackers follow the same phases but with authorisation and documentation.

The 5-Phase Ethical Hacking Methodology

Phase 1: RECONNAISSANCE — Gather information passively and actively
Phase 2: SCANNING & ENUMERATION — Map the attack surface: ports, services, users, shares
Phase 3: VULNERABILITY ANALYSIS — Find specific exploitable weaknesses
Phase 4: EXPLOITATION — Gain access by exploiting vulnerabilities
Phase 5: POST-EXPLOITATION — PrivEsc, lateral movement, persistence, data access
Phase 6: REPORTING — Document all findings with business impact and remediation

Cyber Kill Chain vs MITRE ATT&CK

⛓️ Cyber Kill Chain (Lockheed Martin)

  • Reconnaissance — Target selection, OSINT
  • Weaponisation — Package exploit + payload
  • Delivery — Email, web, USB delivery
  • Exploitation — Trigger the exploit
  • Installation — Install malware, persistence
  • C2 — Establish command and control
  • Actions on Objectives — Exfiltrate, destroy
  • Linear model: defenders break chain at any point

🗂️ MITRE ATT&CK Framework

  • 14 tactics (columns) × 200+ techniques (rows)
  • Reconnaissance, Resource Development, Initial Access
  • Execution, Persistence, Privilege Escalation
  • Defense Evasion, Credential Access, Discovery
  • Lateral Movement, Collection, C2, Exfiltration, Impact
  • Each technique has sub-techniques, examples, detections
  • Universal language for threat intelligence sharing
  • attack.mitre.org — free, industry standard

RULES OF ENGAGEMENT TEMPLATE

📌 Conclusion

Methodology transforms hacking from random exploration into systematic security assessment. The 5-phase model ensures comprehensive coverage. MITRE ATT&CK provides a common language for every technique — defenders can monitor for specific ATT&CK techniques, creating a direct link between offensive and defensive operations.

Topic 132 🔥

Reconnaissance

🔭
⚡ One-Line Summary
Reconnaissance is intelligence gathering before any direct contact — passive recon (no detection) gathers public info; active recon (detectable) probes systems directly to map the target.

Passive vs Active Reconnaissance

🕵️ Passive Recon (Undetectable)

  • No direct contact with target systems
  • Uses publicly available information (OSINT)
  • Zero detection risk
  • Takes: hours to days
  • Sources: Google, LinkedIn, Shodan, DNS records, WHOIS, job postings, GitHub, certificates

🔍 Active Recon (Detectable)

  • Direct interaction with target systems
  • Port scanning, banner grabbing, web crawling
  • Logs created on target — may trigger alerts
  • Must be within scope
  • Tools: nmap, nikto, gobuster, whatweb, wafw00f

📌 Conclusion

Reconnaissance is the most important phase — thorough recon leads to efficient exploitation. Professional attackers spend 80% of their time in recon and only 20% on exploitation. A defender who understands what attackers can discover passively can reduce their attack surface before the attacker even makes contact.

Maintaining Access & Covering Tracks

🕳️
⚡ One-Line Summary
Maintaining access installs persistent backdoors (cron jobs, services, scheduled tasks) so access survives reboots; covering tracks removes evidence (logs, artefacts) to avoid detection — both are key APT tactics.
PERSISTENCE + COVERING TRACKS
━━━ LINUX PERSISTENCE METHODS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Leave a marker file to prove access (don't actually persist unless in scope)
# Report all persistence methods found in the report
⚠️
Ethical Warning: In a real penetration test, covering tracks is generally NOT done — you want your client's SIEM to detect you (to test their visibility). Persistence and track-covering demonstrate the full impact of a breach — showing the client not just that you got in, but that a real attacker would still be there months later. Red Team vs Blue Team

⚔️
⚡ One-Line Summary
Red Team simulates advanced attackers in realistic, long-duration engagements testing detection and response; Blue Team defends, detects, and responds; Purple Team combines both to maximise security improvement.

Team Definitions

TeamRoleGoalToolsMindset
Red TeamAttack (offensive)Simulate APT; find gaps in detection + responseCobalt Strike, C2 frameworks, custom implants"Can we breach the org without being detected?"
Blue TeamDefend (defensive)Detect, investigate, contain, and eradicate threatsSIEM (Splunk), EDR, IDS/IPS, log analysis"How quickly can we detect and stop them?"
Purple TeamCollaborate (both)Red shares TTPs with Blue to improve detection in real-timeAll above + MITRE ATT&CK mapping"How can both teams improve together?"
Yellow TeamBuild (developers)Build secure code, secure pipelines, DevSecOpsSAST, DAST, threat modelling, secure SDLC"Security by design, not by afterthought"

Red Team vs Pen Test

⚔️ Red Team Operation

  • Duration: weeks to months
  • Objective: achieve specific goal (get to crown jewels)
  • Scope: broad — entire organisation
  • Stealth: operate without detection as long as possible
  • Tests: technical + people + physical security
  • Uses custom C2 (Cobalt Strike, Brute Ratel)
  • Report: focuses on detection gaps, not just vulns

🔬 Penetration Test

  • Duration: days to 2 weeks
  • Objective: find as many vulnerabilities as possible
  • Scope: defined targets, specific systems
  • Noise: doesn't matter if detected
  • Technical focus: specific attack surface
  • Uses standard tools (nmap, Metasploit, Burp)
  • Report: comprehensive vulnerability list

              Designed to bypass modern EDR (CrowdStrike, SentinelOne)

Metasploit     — Free, modules for every stage
              Meterpreter shell, post-exploitation modules

Sliver         — Open-source Go-based C2
              mTLS/WireGuard C2 channels, active development

PoshC2         — PowerShell-based C2 (NCSC recommended for red team)
              Implants in memory (PowerShell, .NET), no disk artefacts

⚖️

Ethical hacking without written authorisation is a criminal offence in virtually every country — the Computer Misuse Act (UK), CFAA (US), and equivalent laws apply regardless of intent. "I was just testing" is not a legal defence.

Legal Protections & Documents

LEGAL DOCUMENTATION
Documents you MUST have before any penetration test:

1. WRITTEN AUTHORISATION (most important)
   Signed letter/email from system owner explicitly authorising testing
   "I, [Name], [Title] of [Company] hereby authorise [Tester] to conduct
    penetration testing of the following systems: [scope] during [dates]"

2. STATEMENT OF WORK (SoW)
   Legal contract detailing scope, methodology, deliverables, cost, liability
   Limits liability of tester if unexpected damage occurs

3. NON-DISCLOSURE AGREEMENT (NDA)
   Tester cannot disclose findings to third parties
   Mutual: client cannot disclose tester's tools/methods

4. RULES OF ENGAGEMENT (RoE)
   Technical detail of what is/isn't allowed
   Emergency contacts, stop conditions

# LEGAL GREY AREAS TO KNOW:
→ Bug bounties: you're authorised BUT only within scope. Testing out-of-scope = crime
→ CVE disclosure: finding a vuln in wild software ≠ you can test it on others' systems
→ Third-party systems in scope: you need authorisation from THEM too
→ Accessing data you weren't authorised to see = additional offence even if in scope
→ Overseas testing: laws of BOTH countries apply — some countries criminalise even authorised testing

# Responsible Disclosure:
CVD (Coordinated Vulnerability Disclosure):
1. Researcher finds vulnerability
2. Notifies vendor privately
3. Agrees on disclosure deadline (typically 90 days)
4. Vendor releases patch
5. Researcher publishes advisory
→ This is legal, ethical, and benefits everyone

Full Immediate Disclosure: risky — gives no time to patch before attackers exploit
Selling to brokers (Zerodium): legal grey area; controversial

The legal framework around ethical hacking is absolute — written authorisation is not optional, it's legally required. Every penetration tester must understand the laws of their jurisdiction and the jurisdictions of their clients. When in doubt: ask for written authorisation. Always. No exceptions.
