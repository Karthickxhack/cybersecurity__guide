165 ☁️ Cloud Basics
📖 Definition

Cloud Computing is the delivery of computing services — including servers, storage, databases, networking, software, analytics, and intelligence — over the Internet ("the cloud") to offer faster innovation, flexible resources, and economies of scale.

🎯 Key Characteristics
On-Demand Self-Service Broad Network Access Resource Pooling Rapid Elasticity Measured Service
🌐 Real-Time Example
🏢 Netflix Example

Netflix stores all its movies/shows on AWS (Amazon Web Services). When you press play, data streams from cloud servers — not from Netflix's own physical machines. 🎬

👤 User Request (Browser/App)
🌐 Internet
☁️ Cloud Provider (AWS/Azure/GCP)
🖥️ Virtual Servers & Storage
📦 Response Delivered to User

167 ⚠️ Cloud Threats
📖 Definition

Cloud threats are security risks specific to cloud environments — they exploit the nature of shared infrastructure, remote access, and internet exposure.

#ThreatDescriptionReal Example
1☠️ Data BreachesUnauthorized access to sensitive dataCapital One breach (AWS 2019) — 100M records
2🔓 Insecure APIsExposed or poorly authenticated APIsParler API leak exposed user data 2021
3🧩 MisconfigurationsWrong settings expose resources publiclyExposed S3 buckets (Facebook, 540M users)
4👤 Account HijackingStolen credentials gain admin accessPhishing attack on cloud admin accounts
5🧲 Insider ThreatsMalicious employee misuses accessAWS employee leaked data 2020
6💥 DDoS AttacksOverwhelm cloud resourcesGitHub DDoS 1.35 Tbps — largest recorded
7🔗 Shared TechnologyVulnerabilities in hypervisor/hardwareMeltdown/Spectre CPU flaws

✅ Countermeasures

  • Encrypt data at rest & transit
  • Use MFA on all accounts
  • Apply least-privilege access
  • Regular security audits
  • Enable DDoS protection

❌ Why Threats Succeed

  • Human error (wrong configs)
  • Weak passwords
  • No monitoring alerts
  • Outdated software patches
  • Shadow IT usage

168 🤝 Shared Responsibility
📖 Definition

The Shared Responsibility Model defines which security tasks are handled by the Cloud Provider and which are the Customer's responsibility. "Security OF the cloud vs Security IN the cloud."

LayerProvider ResponsibleCustomer Responsible
🖥️ IaaSPhysical hardware, network, hypervisorOS, apps, data, identity, firewall
🛠️ PaaS+ OS and runtimeApps, data, access control
💻 SaaSAlmost everythingData & user access only
🌐 Cloud Provider = "Security OF the Cloud"
Physical security, hardware, infrastructure
⚖️ Shared Zone (varies by model)
👤 Customer = "Security IN the Cloud"
Data, identity, access, app configs
🔴 Real-Time Example
AWS secures their physical data centers with guards, cameras, and biometrics. YOU are responsible for setting your S3 bucket to "Private" not "Public." If you leave it public, that's YOUR fault. 🏗️

169 🔑 IAM – Identity & Access Management
📖 Definition

IAM (Identity and Access Management) is a framework of policies, technologies, and processes that ensures the right people have the right access to the right resources at the right time.

🧩 IAM Components
👤 Authentication (Who are you?) 🔐 Authorization (What can you do?) 📋 Roles & Policies 🔄 MFA (Multi-Factor Auth) 🔑 SSO (Single Sign-On)
📐 Principle of Least Privilege

Give users/services only the minimum permissions they need — nothing more. A developer who only needs to read logs should NOT have admin rights. 🔒

# AWS IAM Policy Example (JSON) { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": [ "s3:GetObject" // Read only ], "Resource": "arn:aws:s3:::my-bucket/*" } ] } # ↑ This user can ONLY read from my-bucket, nothing else
IAM ConceptDefinitionExample
🙋 UserIndividual identityjohn@company.com
👥 GroupCollection of usersDevelopers, Admins
🎭 RoleTemporary permission setEC2 can access S3
📜 PolicyJSON permission documentAllowS3ReadOnly
🔑 MFASecond auth factorGoogle Authenticator OTP

170 ⚙️ Misconfigurations
📖 Definition

Cloud misconfiguration occurs when cloud services are set up incorrectly — leaving data exposed, permissions too broad, or security features disabled. It is the #1 cause of cloud breaches (Gartner: 99% of cloud failures through 2025 are customer's fault).

Misconfiguration TypeWhat HappensReal Impact
🌐 Public Storage BucketS3/Blob set to publicAnyone can download files
🔓 Overly Permissive IAMAll users = adminMass data deletion/theft
🚫 No EncryptionData stored in plaintextImmediate data exposure
🔥 Open Security Groups0.0.0.0/0 on port 22 (SSH)Anyone can SSH in
📋 No LoggingCloudTrail disabledNo audit trail after breach
🔑 Hardcoded SecretsAPI keys in source codeKeys scraped from GitHub
# BAD: Open security group (AWS CLI) aws ec2 authorize-security-group-ingress \ --group-id sg-12345 \ --protocol tcp \ --port 22 \ --cidr 0.0.0.0/0 # ← Allows SSH from ANYWHERE! # GOOD: Restrict to your IP only aws ec2 authorize-security-group-ingress \ --group-id sg-12345 \ --protocol tcp \ --port 22 \ --cidr 203.0.113.10/32 # ← Only your IP

171 🪣 S3 Bucket Leaks
📖 Definition

Amazon S3 (Simple Storage Service) buckets store files in the cloud. An S3 Bucket Leak occurs when access controls are misconfigured, making private data publicly accessible to anyone on the internet.

💥 Famous S3 Leaks
  • 📘 Facebook – 540M user records (2019)
  • 🚗 Tesla – Internal scripts exposed
  • 🏥 US Military – 1.8B surveillance posts
  • 🎮 Twitch – Source code leaked (2021)
  • 🏦 Capital One – 100M customers
🛡️ How to Secure S3
  • ✅ Block Public Access setting ON
  • ✅ Enable bucket versioning
  • ✅ Enable server-side encryption (SSE)
  • ✅ Use bucket policies (not ACLs)
  • ✅ Enable CloudTrail logging
  • ✅ Use Amazon Macie (auto-detect PII)
👤 Developer creates S3 Bucket
⚠️ Sets ACL = "Public Read"
🔍 Attacker uses S3 Scanner tool
📂 Finds exposed bucket URL
💥 Downloads all data — BREACH!
# Check if S3 bucket is public (AWS CLI) aws s3api get-bucket-acl --bucket my-bucket # Block ALL public access (recommended) aws s3api put-public-access-block \ --bucket my-bucket \ --public-access-block-configuration \ "BlockPublicAcls=true,IgnorePublicAcls=true, BlockPublicPolicy=true,RestrictPublicBuckets=true"

# AWS CloudWatch alarm — CPU spike alert aws cloudwatch put-metric-alarm \ --alarm-name "HighCPU" \ --metric-name CPUUtilization \ --namespace AWS/EC2 \ --threshold 80 \ --comparison-operator GreaterThanThreshold \ --evaluation-periods 2 \ --alarm-actions arn:aws:sns:us-east-1:123456:MyAlert # ↑ Sends SNS notification when CPU > 80% for 2 periods

174 🔄 DevSecOps
📖 Definition

DevSecOps = Development + Security + Operations. It integrates security practices throughout the software development lifecycle (SDLC) rather than adding security at the end. "Shift security left" — catch issues early when they are cheap to fix.

📝 Plan
Threat modeling, security requirements
💻 Code
SAST (Static analysis), peer review
🔨 Build
Dependency scanning, SCA
🧪 Test
DAST, pen testing, IAST
🚀 Deploy
IaC scanning, signed containers
📡 Monitor
SIEM, threat detection, alerts
🔁 Feedback Loop → Back to Plan

✅ Benefits of DevSecOps

  • Faster vulnerability detection
  • Cheaper to fix early-stage bugs
  • Consistent security policies
  • Compliance automation
  • Better team collaboration

❌ Challenges

  • Requires culture change
  • Tool integration complexity
  • Skills gap in security
  • Slows CI/CD if poorly tuned

176 📁 Case Study – Capital One Breach (2019)
📖 Overview

In July 2019, a former AWS employee exploited a misconfigured Web Application Firewall (WAF) on Capital One's AWS infrastructure to steal data of 100 million customers — names, addresses, credit scores, SSNs, and bank accounts.

🏦 Capital One runs on AWS EC2 + S3
⚠️ WAF misconfigured — allows SSRF attack
🔓 Attacker sends request to EC2 metadata endpoint (169.254.169.254)
🔑 Gets IAM credentials from metadata service
📂 Uses credentials to list + download S3 buckets
💥 100M records stolen. $80M+ fine. Criminal charges.
❌ What Went Wrong
  • WAF misconfigured for SSRF
  • EC2 metadata not protected (IMDSv1)
  • IAM role had too many permissions
  • No anomaly detection alerts
✅ Lessons Learned
  • Use IMDSv2 (token-required)
  • Apply least-privilege IAM policies
  • Enable GuardDuty & CloudTrail
  • Regular WAF configuration audits

🎯 Unit 10 Conclusion

Cloud Security is not optional — it is the foundation of every modern digital business. The core insight of this unit: most cloud breaches are not caused by sophisticated hackers breaking encryption — they are caused by misconfigurations, weak access controls, and ignored monitoring alerts.

Master IAM, apply the Shared Responsibility Model, always encrypt, monitor continuously, and embed security into your DevOps pipeline — and you will be ahead of 95% of cloud deployments. ☁️🔐
