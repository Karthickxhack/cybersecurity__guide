Topic 53

Introduction to Cryptography

🔐

Definition

Definition
Cryptography (Greek: kryptós = hidden, gráphein = to write) is the science of securing communication by transforming readable data into an unreadable form and back. It provides the mathematical foundation for all digital security — from HTTPS to Bitcoin to end-to-end encrypted messaging.

Brief History of Cryptography

~58 BC — Caesar Cipher
Julius Caesar shifted letters by 3. "HELLO" → "KHOOR". First documented substitution cipher. Trivially broken today.
1467 — Vigenère Cipher
Polyalphabetic substitution using a keyword. Considered unbreakable for 300 years. Broken by Babbage/Kasiski 1854.
WWII — Enigma Machine
German rotor cipher machine. Cracked by Alan Turing at Bletchley Park using the Bombe machine. Shortened WWII by ~2 years.
1976 — Diffie-Hellman
First public key exchange protocol. Revolutionary: two strangers can create a shared secret over a public channel.
1977 — RSA
Rivest-Shamir-Adleman. First practical public-key cryptosystem. Still used today in TLS certificates.
2001 — AES
NIST adopts AES (Rijndael). Replaces DES as the global standard. 128/192/256-bit keys. Unbroken to this day.
2009 — Bitcoin
Satoshi Nakamoto uses SHA-256 and ECDSA. Blockchain — cryptography applied to decentralised finance.
2022+ — Post-Quantum
NIST selects CRYSTALS-Kyber and CRYSTALS-Dilithium. Quantum computers threaten RSA/ECC — PQC is the response.

Goals of Cryptography (CIA+ Model)

🔒
Confidentiality
Only authorised parties can read the data. Achieved by encryption.
✅
Integrity
Data has not been tampered with in transit. Achieved by hash functions & MACs.
🪪
Authentication
Verifying identity of sender. Achieved by digital signatures & certificates.
📜
Non-repudiation
Sender cannot deny sending the message. Achieved by digital signatures.
🔑
Key Exchange
Securely establishing a shared key over public channel. Diffie-Hellman, ECDHE.
📵
Anonymity
Hiding identity of communicating parties. Tor, zero-knowledge proofs.

Cryptography vs Steganography vs Obfuscation

MethodApproachExampleSecurity Level
CryptographyTransforms data to ciphertext — presence is known, content hiddenAES-encrypted fileHigh (with strong keys)
SteganographyHides data inside innocent carrier — existence is hiddenSecret message in image LSBsMedium (easily found if suspected)
ObfuscationMakes code/data harder to read without truly hiding itMinified JavaScriptLow (reversible by expert)

📌 Conclusion

Cryptography is the mathematical bedrock of all cybersecurity. Without it, there is no HTTPS, no secure messaging, no digital banking, no blockchain. Every cybersecurity professional must understand its principles — even if they don't implement the mathematics themselves.

Encryption & Decryption

🔄

Core Concepts

Definition
Encryption converts plaintext (readable) into ciphertext (unreadable) using an algorithm and a key. Decryption reverses this using the same or related key. The security relies on the key, not the secrecy of the algorithm (Kerckhoffs's Principle).

Types of Encryption

🔑 Symmetric Encryption

  • Same key for encryption AND decryption
  • Fast — suitable for large data
  • Problem: how to securely share the key?
  • Examples: AES, DES, 3DES, ChaCha20
  • Used for: file encryption, disk encryption, bulk data

🗝️ Asymmetric Encryption

  • Public key encrypts, private key decrypts
  • Slow — only for small data / key exchange
  • Solves the key distribution problem
  • Examples: RSA, ECC, ElGamal, Diffie-Hellman
  • Used for: TLS handshake, digital signatures, key exchange

Kerckhoffs's Principle

💡
Kerckhoffs's Principle (1883): A cryptosystem should be secure even if everything about the system, except the key, is public knowledge. This is why AES, RSA, and all modern algorithms are publicly documented — security comes from key secrecy, not algorithm secrecy. "Security through obscurity" alone is never sufficient.

Encryption Modes (Block Ciphers)

ENCRYPTION MODES
ECB (Electronic Codebook)   — INSECURE
Each block encrypted independently. Same plaintext → same ciphertext.
Shows patterns! Never use ECB for images or structured data.

CBC (Cipher Block Chaining)  — ACCEPTABLE
Each block XORed with previous ciphertext before encrypting.
Requires IV (Initialization Vector). Sequential, cannot parallelise.
Vulnerable to padding oracle attacks (POODLE on SSL3)

CTR (Counter Mode)          — GOOD
Encrypts incrementing counter value, XORs with plaintext.
Parallelisable. IV/nonce must NEVER be reused with same key!
Nonce reuse = catastrophic — XOR of two ciphertexts reveals XOR of plaintexts

GCM (Galois/Counter Mode)   — RECOMMENDED ✓
CTR + authentication tag (GHASH). Provides both encryption AND integrity.
Used in TLS 1.3: AES-256-GCM. Authenticated Encryption with Associated Data (AEAD).

# Python example (AES-256-GCM with cryptography library):
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os
key = os.urandom(32)    # 256-bit key
nonce = os.urandom(12)  # 96-bit nonce — NEVER REUSE
aes = AESGCM(key)
ciphertext = aes.encrypt(nonce, b"Secret message", b"additional data")
plaintext  = aes.decrypt(nonce, ciphertext, b"additional data")
🔬 Exercise
  1. Why is ECB mode dangerous? Find the famous "ECB penguin" image online — what does it demonstrate?
  2. What is an IV/nonce, and what happens in CTR mode if you reuse the same nonce with the same key?
  3. What additional security property does GCM provide over CTR mode?
📌 Conclusion

Encryption mode matters as much as the algorithm. AES in ECB is dangerous. AES in GCM is excellent. Always use authenticated encryption (AEAD) modes like GCM or ChaCha20-Poly1305 — they protect both confidentiality and integrity.

Topic 55

Symmetric Encryption

🔑

Definition & Algorithms

Definition
Symmetric encryption uses the same key for both encryption and decryption. It is fast and efficient for encrypting large amounts of data but requires a secure way to share the key between parties (the key distribution problem).

Algorithm Reference

AES
BLOCK CIPHER · NIST STANDARD
Advanced Encryption Standard. 128/192/256-bit keys. 10/12/14 rounds. Substitution-permutation network. Used everywhere.
✓ SECURE
ChaCha20
STREAM CIPHER · IETF RFC 8439
20-round stream cipher. Very fast on devices without AES hardware acceleration. Used in TLS 1.3, WireGuard, Android.
✓ SECURE
3DES
BLOCK CIPHER · LEGACY
Triple DES: Encrypt-Decrypt-Encrypt with 2-3 keys. Very slow. 112-bit effective security. Legacy financial systems.
⚠ LEGACY
DES
BLOCK CIPHER · BROKEN
56-bit key. Cracked in 22 hours (1999) using brute force. Predecessor to AES. Never use.
✗ BROKEN
RC4
STREAM CIPHER · BROKEN
Once used in WEP and TLS. Statistical biases in keystream. Prohibited by RFC 7465 in TLS. Never use.
✗ BROKEN
Blowfish
BLOCK CIPHER · LEGACY
Bruce Schneier, 1993. 32–448-bit keys. Still used in bcrypt password hashing (intentionally slow). 64-bit block = SWEET32 risk.
⚠ LEGACY

AES Inner Workings

AES ROUND STRUCTURE
# AES operates on a 4×4 byte "state" matrix (128-bit block)
# Each round applies 4 transformations:

Round Key Addition:  XOR state with round key (derived from main key)
SubBytes:           Replace each byte using 16×16 S-Box (non-linear)
ShiftRows:          Cyclically shift each row left by 0,1,2,3 positions
MixColumns:         Multiply each column by a fixed polynomial (diffusion)

# AES-128: 10 rounds | AES-192: 12 rounds | AES-256: 14 rounds
# Final round omits MixColumns step

# Why AES is secure:
→ Confusion (SubBytes) — non-linear, hides relationship between key and ciphertext
→ Diffusion (ShiftRows + MixColumns) — change 1 input bit → ~50% of output bits change
→ Key schedule — 128-bit key expands to 1408 bits of round key material

# Security: No practical attack faster than brute force
# Brute force AES-128: 2^128 operations ≈ 10^38 years at 1 billion ops/sec
# AES-256: 2^256 ≈ more than atoms in observable universe

Key Size vs Security

3DES112-bit eff.Theoretically huge, but slow & SWEET32Legacy only

The key distribution problem is solved by using asymmetric encryption to securely exchange the symmetric key — giving us the best of both worlds.

Topic 56

Asymmetric Encryption

🗝️

Definition

Definition
Asymmetric encryption uses a mathematically linked key pair: a public key (shared openly) and a private key (kept secret). Data encrypted with the public key can only be decrypted with the private key, and vice versa. This solves the key distribution problem entirely.

Two Use Cases: Encryption vs Signing

ASYMMETRIC KEY PAIR USAGE
USE CASE 1: ENCRYPTION (Confidentiality)
Alice wants to send Bob a secret:
  Bob publishes his PUBLIC KEY
  Alice encrypts message with Bob's PUBLIC KEY
  Only Bob can decrypt with his PRIVATE KEY
  → Anyone can send to Bob, only Bob can read it

USE CASE 2: DIGITAL SIGNATURE (Authentication + Non-repudiation)
Alice wants to prove she sent a message:
  Alice signs message with her PRIVATE KEY
  Anyone verifies signature with Alice's PUBLIC KEY
  → Only Alice could have created it (she holds private key)
  → Non-repudiation: Alice cannot deny signing

Memory aid:
  Encrypt with PUBLIC   → Recipient's private key decrypts  (send secret TO someone)
  Sign with PRIVATE      → Anyone's public key verifies       (prove it's FROM you)

RSA — How It Works

RSA MATHEMATICS
RSA Security is based on: Integer Factorisation Problem
It's easy to multiply two large primes → hard to factor the result

Key Generation:
1. Choose two large primes: p = 61, q = 53   (tiny example!)
2. Compute n = p × q = 3233                  (modulus — public)
3. Compute φ(n) = (p-1)(q-1) = 3120           (kept secret)
4. Choose e coprime to φ(n): e = 17            (public exponent)
5. Find d such that e × d ≡ 1 (mod φ(n)): d = 2753 (private exponent)

Public Key: (e=17, n=3233)    Private Key: (d=2753, n=3233)

Encrypt:  ciphertext = plaintext^e mod n
Decrypt:  plaintext  = ciphertext^d mod n

# Real RSA uses primes with 2048–4096 bits (617–1234 decimal digits)
# Breaking RSA-2048 = factoring a 617-digit number
# Best classical algorithm: Number Field Sieve — takes millions of years
# Quantum threat: Shor's algorithm factors in polynomial time!
# → RSA will be BROKEN by quantum computers (motivation for PQC)

ECC — Elliptic Curve Cryptography

⚡
ECC (Elliptic Curve Cryptography) achieves the same security as RSA but with much smaller key sizes. Security is based on the Elliptic Curve Discrete Logarithm Problem (ECDLP). A 256-bit ECC key is equivalent in strength to a 3072-bit RSA key. Used in Bitcoin (secp256k1), TLS 1.3 (X25519), and modern SSH.

🔬 Exercise
  1. Bob wants to receive encrypted messages. Should he share his public key or private key? What happens if he accidentally publishes his private key?
  2. Why is RSA used for key exchange rather than encrypting large files?
  3. Why does a 256-bit ECC key provide more security than a 256-bit RSA key?
📌 Conclusion

Asymmetric encryption solves the fundamental key distribution problem but is too slow for bulk data. In practice, asymmetric encryption is used to securely exchange a symmetric key, which then encrypts the actual data — this hybrid approach is used in TLS, PGP, and virtually all secure protocols.

Topic 57

Hash Functions

#️⃣

Definition

Definition
A cryptographic hash function is a one-way mathematical function that takes any-length input and produces a fixed-length output (the digest or hash). It is deterministic, irreversible, and a tiny change in input produces a completely different output (avalanche effect). It is NOT encryption — you cannot recover the original data from a hash.

Properties of Secure Hash Functions

HASH PROPERTIES + EXAMPLES
1. Deterministic:
   SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
   SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
   Same input → always same output

2. Avalanche Effect:
   SHA-256("hello") = 2cf24dba5fb0a30e...
   SHA-256("Hello") = 185f8db32921bd46...
   One character change → completely different hash (50% of bits flip)

3. Fixed Output Size (regardless of input size):
   SHA-256(1 byte)  → 256 bits
   SHA-256(1 GB)    → 256 bits

4. Pre-image Resistance (One-way):
   Given hash H, cannot find input M such that hash(M) = H
   Cannot reverse SHA-256 to find original password

5. Collision Resistance:
   Cannot find two different inputs M1 ≠ M2 where hash(M1) = hash(M2)
   MD5 collisions found 2004. SHA-1 collision found 2017 (SHAttered attack).

6. Avalanche / Strict Bit Sensitivity:
   Every input bit influences every output bit

Real-World Uses of Hashing

🔐
Password Storage
Never store plaintext! Store hash(password+salt). Attacker gets hash, cannot reverse it.
📁
File Integrity
SHA-256 checksum verifies download hasn't been tampered with. Linux ISO distributions publish hashes.
⛓️
Blockchain
SHA-256 links blocks. Each block contains hash of previous block — tamper one, break the chain.
✍️
Digital Signatures
Sign the hash of a document (not the document itself). Much faster than signing the full document.
🔗
HMAC (MAC)
Hash + secret key = Message Authentication Code. Proves both integrity AND sender identity.
🔑
Key Derivation
PBKDF2, scrypt, Argon2: derive cryptographic keys from passwords using iterated hashing.
🔬 Exercise
  1. Hash "password123" using the concept of SHA-256. Now hash "password124". What property ensures these produce completely different outputs?
  2. A database leaks hashed passwords using unsalted MD5. Why can the attacker crack them quickly using rainbow tables?
  3. What is HMAC and why does it provide more security than a plain hash for message authentication?
📌 Conclusion

Hash functions are the DNA fingerprint of digital data. MD5 and SHA-1 are cryptographically broken and must never be used for security purposes. Use SHA-256+ for data integrity, bcrypt/Argon2 for passwords, and HMAC for message authentication.

Topic 58

Digital Signatures

✍️

Definition

Definition
A digital signature is a cryptographic mechanism that provides authentication (proves who signed), integrity (proves data wasn't modified), and non-repudiation (signer cannot deny signing). Created using the signer's private key, verified using their public key.

How Digital Signatures Work

Alice writes document: "Transfer £10,000 to Bob"
1️⃣ Hash the document: SHA-256(document) = hash_value
2️⃣ Encrypt hash with Alice's PRIVATE KEY = Signature
3️⃣ Send: Document + Signature → Bob
Bob receives document + signature
4️⃣ Hash the received document → new_hash
5️⃣ Decrypt signature with Alice's PUBLIC KEY → original_hash
6️⃣ Compare: new_hash == original_hash? ✅ Valid Signature

Digital Signature Algorithms

DSADiscrete logarithm1024–3072-bitLegacy — deprecated by NIST 2023Avoid
ECDSAElliptic curve DL256–521-bitTLS, Bitcoin, code signingRecommended
EdDSA (Ed25519)Edwards curve255-bitSSH, Signal, Tor, WireGuardBest
Dilithium (CRYSTALS)Lattice (Module LWE)1312–2592-bitPost-quantum standard (NIST)Future-proof

Real-World Applications

📦
Code Signing
Windows/macOS only run software signed by verified developers. Prevents malware distribution.
📧
Email (S/MIME, PGP)
Sign emails to prove sender identity. Prevents email spoofing at application level.
⛓️
Blockchain
Bitcoin uses ECDSA to authorise transactions. Only the holder of the private key can spend the funds.
📜
Document Signing
DocuSign, Adobe Sign. Legally binding digital signatures on contracts, leases, government documents.
🌐
TLS Certificates
CA signs server certificate. Browser verifies signature using CA's public key to authenticate the server.
🔐
SSH Authentication
Instead of passwords, SSH uses key pairs. Server verifies client has private key matching authorised public key.
🔬 Exercise
  1. Why do digital signatures hash the document first, rather than encrypting the whole document with the private key?
  2. Alice signs a document. Bob receives it and verifies the signature. What three security properties does this provide Bob?
  3. An attacker modifies the document but doesn't change the signature. What happens when Bob verifies it?
📌 Conclusion

Digital signatures are the legal signature of the digital world. They prove who created data and that it hasn't been modified. Used in TLS, software distribution, blockchain, and government systems — digital signatures underpin the trust model of the entire internet.

Topic 59

Public Key Infrastructure (PKI)

🏛️

Definition

Definition
PKI (Public Key Infrastructure) is a framework of policies, procedures, hardware, software, and people that manages the creation, distribution, storage, and revocation of digital certificates. It solves the public key authentication problem: how do you know a public key really belongs to who claims to own it?

PKI Hierarchy

TRUST CHAIN
                    ┌────────────────────────────────┐
                    │    ROOT CA (Certificate Authority)   │
                    │    Self-signed certificate           │
                    │    e.g. "DigiCert Global Root G2"    │
                    │    Pre-installed in OS & browsers    │
                    └──────────────┬─────────────────────┘
                                   │ issues certificate to ▼
                    ┌──────────────▼─────────────────────┐
                    │   INTERMEDIATE CA                      │
                    │   Signed by Root CA                  │
                    │   Issues end-entity certificates     │
                    │   Protects Root CA (kept offline)    │
                    └──────────────┬─────────────────────┘
                                   │ issues certificate to ▼
                    ┌──────────────▼─────────────────────┐
                    │   END-ENTITY CERTIFICATE              │
                    │   website: google.com                │
                    │   Signed by Intermediate CA          │
                    │   Contains: public key + identity    │
                    └────────────────────────────────────┘

# Verification process (browser validates google.com cert):
1. Browser receives google.com's certificate
2. Checks it's signed by a trusted Intermediate CA
3. Checks Intermediate CA is signed by trusted Root CA
4. Root CA is in browser's trusted store → CHAIN OF TRUST established
5. Verify not expired, not revoked (OCSP/CRL check)
6. Verify hostname matches certificate CN/SAN

Trust Stores & Attacks

⚠️
DigiNotar 2011 (Real-world PKI Attack): Dutch CA DigiNotar was hacked. Attackers issued fraudulent certificates for google.com, mozilla.org, and other major sites — allowing MitM attacks on millions of Iranian users. Browsers immediately revoked trust in DigiNotar. The CA went bankrupt. Lesson: a compromised CA destroys the entire trust chain for that CA.
🔬 Exercise
  1. Why do browsers ship with ~150 pre-installed Root CAs? What is the risk of this design?
  2. Explain the "chain of trust" concept. Why are Root CAs kept offline?
  3. What is certificate pinning, and how does it protect against rogue CA attacks?
📌 Conclusion

PKI is the trust backbone of the internet. Every HTTPS connection, every code-signed application, every S/MIME email relies on it. Its Achilles heel is the CA — a single compromised CA can undermine trust for the entire internet. Certificate Transparency logs (crt.sh) now make unauthorised certificate issuance detectable.

# TLS 1.2 weak suites to DISABLE:
TLS_RSA_WITH_RC4_128_MD5         ← RC4 broken + MD5 broken
TLS_RSA_WITH_3DES_EDE_CBC_SHA    ← 3DES + CBC = SWEET32 + BEAST
TLS_DHE_RSA_WITH_DES_CBC_SHA     ← DES broken

Forward Secrecy

🔑
Perfect Forward Secrecy (PFS): If the server's private key is compromised in the future, it cannot be used to decrypt previously recorded traffic. Achieved by generating a fresh ephemeral ECDH key for each session. TLS 1.3 mandates forward secrecy. TLS 1.2 without ECDHE (using static RSA key exchange) does NOT have PFS — compromised server key = all past traffic decryptable.
🔬 Exercise
  1. What is the difference between TLS 1.2 and TLS 1.3 in terms of handshake round trips and security?
  2. Explain Perfect Forward Secrecy. Why did the NSA PRISM program make PFS suddenly critical for all websites?
  3. Decode this cipher suite: TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256
📌 Conclusion

TLS 1.3 is the gold standard for transport security. It's faster (1-RTT), drops all broken ciphers, and mandates forward secrecy. Deploying TLS 1.3 with HSTS and strong cipher suites is the single most impactful security upgrade for any web service.

Attack Scenario: Rainbow Table Attack

RAINBOW TABLE ATTACK + DEFENCE
# WITHOUT salt (VULNERABLE):
Database stores: SHA-256("password123") = ef92b778...

Attacker has rainbow table:
  "password123" → ef92b778...  ← INSTANT MATCH!
  "hello123"    → b94f6f125...
  "letmein"     → 1c8bfe8f...

→ 14 million common passwords cracked in <1 second!

# WITH SALT (SECURE):
Random salt generated per user: salt = "xK9mP3qR"
Store: salt + SHA-256(salt + "password123") = db record

Attacker's rainbow table: useless! Would need to rebuild entire
table for every unique salt. 10 million salts = 10 million tables.

import argon2

# Argon2: salt built-in, memory-hard, parallelism-hard → GPU cracking much harder

Topic 64

Salting & Hashing

🧂

The Problem with Unsalted Hashes

SALTING DEEP DIVE
# Problem: same password → same hash (everywhere, for all users)
User A: password = "letmein" → MD5 = 0d107d09f5bbe40cade3de5c71e9e9b7
User B: password = "letmein" → MD5 = 0d107d09f5bbe40cade3de5c71e9e9b7
→ Crack once = crack everyone with same password
→ Rainbow tables precompute BILLIONS of these mappings

# Solution: Salt — unique random value added per user BEFORE hashing
User A: salt = "aK9xPQ3m", hash = SHA-256("aK9xPQ3m" + "letmein") = 8f43e2...
User B: salt = "zR7yTL2n", hash = SHA-256("zR7yTL2n" + "letmein") = c91ab4...
→ Same password, completely different hashes → rainbow tables useless

Correct storage format (store in DB):
{
  "username": "alice",
  "salt": "aK9xPQ3m",        ← stored in cleartext (not a secret!)
  "hash": "8f43e2ca..."       ← salted hash
}

Login verification:
1. Retrieve salt for username from DB
2. Compute: SHA-256(stored_salt + entered_password)
3. Compare to stored hash
4. Match → authenticated

# Modern best practice: bcrypt/scrypt/Argon2 build salt IN and are slow:
bcrypt: $2b$12$LQv3c1yqBWVHxkd0LHAkCOYz6TtxMQJqhN8/LewdBPj/VK0v7Cquu
         └─┘└─┘└───────────────────────────────────────────────────────┘
         alg cost     22-char salt + 31-char hash (all base64)
# cost=12 means 2^12=4096 iterations → ~100ms per check → brute force: 10 hashes/sec (not 10 billion!)

End-to-End Encryption (E2EE)

SIGNAL PROTOCOL OVERVIEW
Signal Protocol (Double Ratchet + X3DH):
Used by: Signal, WhatsApp, Facebook Messenger (secret chats)

X3DH (Extended Triple Diffie-Hellman) — Initial key agreement:
  Alice and Bob exchange 4 Diffie-Hellman calculations using
  long-term, medium-term, and one-time prekeys
  → Derives initial shared secret without being online simultaneously

Double Ratchet Algorithm — Ongoing messages:
  Each message uses a NEW symmetric key derived from previous key
  → Forward secrecy: compromise of today's key ≠ read past messages
  → Break-in recovery: new keys every message ≠ read future messages
  
Result: Even Signal/WhatsApp servers CANNOT read your messages
Law enforcement can only get metadata (who messaged whom, when)
Message content: mathematically inaccessible without device keys

Warning: E2EE only protects in transit. Endpoint security matters:
→ Compromised phone = all messages visible (Pegasus spyware)
→ Cloud backup of messages (iCloud, Google Drive) may be unencrypted

SSH Key Authentication (Best Practice)

SSH SECURITY
# Generate Ed25519 SSH key pair (modern, secure):
ssh-keygen -t ed25519 -C "alice@company.com" -f ~/.ssh/id_ed25519
# Creates: id_ed25519 (PRIVATE — never share) + id_ed25519.pub (PUBLIC)

# Copy public key to server:
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com

# Secure SSH server config (/etc/ssh/sshd_config):
PermitRootLogin no          # Never allow root SSH
PasswordAuthentication no   # Keys only — no passwords
PubkeyAuthentication yes
AllowUsers alice bob        # Whitelist specific users
Port 2222                   # Non-standard port (reduces automated scans)
MaxAuthTries 3
LoginGraceTime 20
Protocol 2                  # SSH v2 only
HostKeyAlgorithms ssh-ed25519,rsa-sha2-512
KexAlgorithms curve25519-sha256

# Authentication flow:
1. Client sends public key to server
2. Server checks authorized_keys — is this key allowed?
3. Server sends random challenge encrypted with client's public key
4. Client decrypts with private key, signs challenge → sends back
5. Server verifies signature with public key → authenticated!
# Private key NEVER transmitted — only proof of possession

Case Study: HTTPS Security

🔍

Full HTTPS Connection Analysis

COMPLETE HTTPS ANALYSIS
SCENARIO: Alice connects to https://bank.example.com
─────────────────────────────────────────────────────

STEP 1: DNS Resolution
  nslookup bank.example.com → 203.0.113.50
  (Security: DNS poisoning could redirect here — DNSSEC mitigates)

STEP 2: TCP Connection
  SYN → SYN-ACK → ACK  to 203.0.113.50:443

STEP 3: TLS 1.3 Handshake (1 round trip)
  → ClientHello: TLS 1.3, ECDHE key share (X25519), cipher prefs
  ← ServerHello: TLS_AES_256_GCM_SHA384 chosen
  ← Certificate: CN=bank.example.com, Issuer=DigiCert
  ← CertificateVerify: ECDSA signature over handshake
  ← Finished: HMAC of all handshake data
  → Finished

  Alice's browser verifies:
  ✓ Certificate signed by trusted CA (DigiCert in trust store)
  ✓ CN/SAN matches bank.example.com
  ✓ Cert not expired (not before: 2024-01, not after: 2025-01)
  ✓ OCSP staple not revoked
  ✓ In Certificate Transparency logs
  ✓ HSTS header forces HTTPS for 1 year

STEP 4: Encrypted Application Data
  POST /login HTTP/2
  [Encrypted with AES-256-GCM, key: ephemeral ECDHE session key]
  Body: {"username":"alice","password":"SecurePass#99"}
  (All plaintext above is invisible to network observer)

STEP 5: Server verifies login, returns session cookie
  Set-Cookie: session=jwt_token; HttpOnly; Secure; SameSite=Strict
  (Cookie protected in transit by TLS + protected in browser by HttpOnly)

WHAT AN ATTACKER ON THE SAME WiFi CAN SEE:
  ✓ IP: 203.0.113.50
  ✓ SNI (Server Name Indication): bank.example.com
  ✓ Approximate data size (not content)
  ✓ Timing of connections
  CANNOT SEE: URL paths, HTTP headers, POST body, cookies, passwords!

📌 Conclusion

The best cipher in the world, implemented incorrectly, provides zero security. Cryptographic failures in real deployments are almost never caused by mathematical breaks of AES or SHA-256 — they come from implementation errors, weak modes, short keys, reused nonces, and missing validation. Use well-known libraries. Read the docs. Default to conservative choices.

Other Future Trends

🔮
Homomorphic Encryption
Compute ON encrypted data without decrypting. Cloud processes your private data without seeing it. Currently 1000x slow — improving rapidly.

Zero-Knowledge Proofs
Prove you know something without revealing what. e.g. Prove age ≥ 18 without revealing birthdate. Used in zk-SNARKs (Ethereum) and privacy coins.
🔑
Passkeys (FIDO2)

🌐
Hybrid PQC + Classic
TLS 1.3 now supports hybrid key exchange (X25519 + Kyber). Classical + PQC together: secure against both classical and quantum attacks.
⚡
Quantum Key Distribution
Use quantum physics (photon polarisation) to detect any eavesdropping on key exchange. Theoretically unbreakable. Range-limited (~100km fibre).
🧮
Threshold Signatures
Split private key across N parties; need M to sign (M-of-N). Used in crypto custody, cloud HSMs. Eliminates single-point-of-key-failure.

Password entropy:    H = L × log₂(N)   (L=length, N=charset size)
Birthday bound:       ~√(2^n) = 2^(n/2) operations for collision
Brute force:         2^(key_bits) operations needed

Algorithm Selection Guide:
Need bulk encryption?          → AES-256-GCM or ChaCha20-Poly1305
Need asymmetric encryption?    → RSA-4096 or ECDSA P-256 / Ed25519
Need a hash?                   → SHA-256 / SHA-3 / BLAKE3
Need password storage?         → Argon2id (PHC winner)
Need key exchange?             → ECDHE X25519 (TLS 1.3)
Need email signing?            → S/MIME with RSA-2048 or GPG
Need quantum-safe?             → Kyber-1024 (KEM) + Dilithium3 (sig)
NEVER use:                     → DES, RC4, MD5 (security), SHA-1, ECB mode

TLS 1.3 cipher suite breakdown:
TLS_ECDHE_[auth]_WITH_[cipher]_[keysize]_[mode]_[hash]
  ECDHE = key exchange with forward secrecy
  auth  = server certificate algorithm
  cipher = bulk encryption (AES / ChaCha20)
  mode  = GCM or Poly1305 (both AEAD)
  hash  = PRF hash (SHA-384 / SHA-256)

Trust chain: Your browser → Root CA → Intermediate CA → Website Cert
Sign:        Private key  →  Verify: Public key
Encrypt:     Recipient's public key  →  Decrypt: Recipient's private key

📌 Unit 4 Final Takeaway

Cryptography is the mathematical foundation that makes all digital trust possible. It is simultaneously ancient (Caesar cipher) and cutting-edge (post-quantum lattice cryptography). The algorithms we use today — AES, SHA-256, RSA, ECDSA — are mathematically sound. The failures always happen at the implementation layer: wrong modes, broken algorithms, poor key management, hardcoded secrets. Master the principles. Use vetted libraries.
