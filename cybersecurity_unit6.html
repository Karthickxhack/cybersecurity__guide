Network Security Basics

🛡️

Definition & Core Principles

Definition
Network security is the practice of protecting the usability, integrity, and safety of a network and its data by implementing hardware and software solutions that prevent unauthorised access, misuse, modification, or denial of the network and its resources.

CIA Triad Applied to Networks

🔒
Confidentiality
Prevent unauthorised reading of network traffic. TLS, VPN, network encryption.
✅
Integrity
Ensure traffic is not tampered with in transit. IPSec, HMAC, digital signatures.
⚡
Availability
Ensure network services remain accessible. Anti-DDoS, redundancy, QoS, rate limiting.
🪪
Authentication
Verify identity of devices and users. 802.1X, MFA, certificates, RADIUS.
🎯
Authorisation
Control what authenticated users can access. ACLs, firewall rules, VLANs, RBAC.
📋
Non-repudiation
Audit logs prove who did what, when. SIEM, logging, network flow records.

IDS & IPS

🚨

IDS vs IPS

🔍 IDS — Intrusion Detection System

  • Monitors traffic — alerts only, does NOT block
  • Placed out-of-band (copy of traffic via SPAN port)
  • No impact on network performance
  • Risk: alert fatigue — thousands of alerts/day
  • Network IDS (NIDS) or Host IDS (HIDS)
  • Tools: Snort (IDS mode), Zeek, Suricata, OSSEC

🛡️ IPS — Intrusion Prevention System

  • Monitors AND blocks malicious traffic inline
  • Placed in-line — all traffic passes through it
  • Can block, drop, reset connections in real-time
  • Risk: false positives block legitimate traffic
  • Network IPS (NIPS) or Host IPS (HIPS)
  • Tools: Snort inline, Suricata, Cisco Firepower

Detection Methods

MethodHow It WorksProsCons
Signature-basedMatch traffic against database of known attack patternsFast, accurate for known attacks, low false positivesZero-days invisible, must update signatures constantly
Anomaly-basedBuild baseline of normal traffic, alert on deviationsDetects unknown attacks, zero-daysHigh false positives until baseline mature, complex tuning
Policy-basedAlert when predefined security policy is violatedSimple, predictableMust manually define every policy, misses novel attacks
Reputation-basedBlock known malicious IPs/domains via threat intelligence feedsBlocks C2 communications instantlyAttackers rotate IPs, feeds lag attacker speed

Packet Sniffing

👃

Definition

Definition
Packet sniffing (network sniffing) is the capture and analysis of data packets traversing a network. Legitimate use: network troubleshooting, performance analysis. Malicious use: credential theft, session hijacking, and data interception. Any unencrypted protocol is fully readable to a sniffer.

Promiscuous Mode & Monitor Mode

🔌 Promiscuous Mode (Wired)

  • NIC accepts ALL packets, not just its own
  • Normally NIC discards packets not destined for it
  • Required for passive sniffing on wired networks
  • Enable: ip link set eth0 promisc on
  • Only captures traffic on same switch/hub segment

📡 Monitor Mode (WiFi)

  • Captures ALL WiFi frames in range (not just your network)
  • Used for WiFi auditing, WPA handshake capture
  • Enable: airmon-ng start wlan0 → wlan0mon
  • Requires compatible wireless card (Alfa AWUS036ACH)
  • Can capture all network beacons and probe requests
🔬 Exercise

📌 Conclusion

Packet sniffing is why HTTPS is non-negotiable. Every plaintext protocol (HTTP, FTP, Telnet, SMTP without STARTTLS) is completely readable by any network observer. Wireshark and tcpdump are indispensable both for defenders (troubleshooting) and attackers (credential capture).

DoS & DDoS Attacks

💥

Definitions

💢 DoS (Denial of Service)

  • Single attacker floods a target
  • Exhausts: bandwidth, CPU, memory, or connection table
  • Easier to block — single source IP
  • Tools: hping3, nping, slowloris
  • Impact: target service unavailable

🌐 DDoS (Distributed DoS)

  • Thousands/millions of sources simultaneously
  • Traffic from botnet, amplification, or rented
  • Very hard to block — traffic looks legitimate
  • Volume: 1Tbps+ attacks recorded (Akamai/Cloudflare)
  • Requires CDN, scrubbing, or ISP-level mitigation

Zero Trust Security

🔄

Zero Trust Principles

Zero Trust Philosophy
"Never Trust, Always Verify" — Zero Trust abandons the traditional perimeter model (trust everything inside the network). Instead: verify every access request regardless of network location, enforce least privilege, assume breach at all times. Popularised by Google's BeyondCorp (2009) and mandated by US Executive Order 14028 (2021).

Zero Trust vs Perimeter Security

🏰 Traditional Perimeter (Castle & Moat)

  • "Trust but verify" inside the network
  • VPN = drawbridge → once in, full access
  • Flat internal network — lateral movement easy
  • Implicit trust for internal IP addresses
  • Single breach = full internal access
  • Fails against insider threats and supply chain

🔄 Zero Trust (BeyondCorp Model)

  • "Never trust, always verify" everywhere
  • Identity-centric — user + device + context = access
  • Micro-segmentation — every service has its own perimeter
  • MFA required for every resource access
  • Continuous verification — session re-verified periodically
  • Limits blast radius if credential is compromised

Zero Trust Architecture Components

User + Device requests access to resource
Identity Verification: MFA + SSO (Okta, Azure AD, Duo)
Device Compliance Check: Is device patched? EDR installed? Certificate valid?
Context Analysis: Unusual location? Time? Risk score? Behavioural baseline?
Policy Decision Point: Allow, Deny, or Step-up auth?
Access granted to SPECIFIC resource only (not full network)
Continuous monitoring: Re-verify periodically, log all actions

The Anatomy of the Target Breach

TARGET BREACH 2013 — FULL ATTACK CHAIN
Outcome:  40M payment cards + 110M personal records stolen
Cost:     $292M total — $18.5M settlement + $162M cybersecurity costs
Cause:    Third-party HVAC vendor compromise

STEP 1: Phishing the HVAC Vendor (Fazio Mechanical)
  → Attacker sends phishing email to Fazio (HVAC contractor for Target)
  → Employee opens malicious attachment → Citadel malware installed
  → Attacker steals Fazio's credentials for Target's vendor portal

STEP 2: Pivoting to Target's Network
  → Fazio had remote access to Target's HVAC/refrigeration monitoring
  → Attacker logs in with stolen Fazio credentials
  → Target's network: HVAC systems on SAME network segment as POS systems!
  CRITICAL FAILURE: No network segmentation between vendor access and POS

STEP 3: Lateral Movement to POS Systems
  → Attacker discovers 1,800+ POS terminals on the network
  → Moves laterally — no internal firewall to stop this
  → Installs custom BlackPOS RAM-scraping malware on POS devices

STEP 4: Data Exfiltration
  → BlackPOS malware scrapes card data from RAM (before encryption)
  → Sends data to internal staging servers
  → Data exfiltrated to external FTP servers in Russia
  → Active from Nov 27 – Dec 15, 2013 (Black Friday to peak shopping!)
  → FireEye alerted Target's team — alerts IGNORED!

LESSONS LEARNED:
  1. Network segmentation: HVAC ≠ POS network  [MISSING]
  2. Third-party access review: what CAN vendors access?  [MISSING]
  3. Least privilege: vendors get only what they need  [MISSING]
  4. Respond to security alerts: FireEye fired — no action taken  [IGNORED]
  5. Internal firewall between segments  [MISSING]
  6. Payment card data should be encrypted at POS  [MISSING]

Quick Reference Glossary

FirewallFilters traffic by rules. Stateful tracks state. NGFW = L7 DPI. Default-deny policy.
IDSDetects attacks, alerts only. Out-of-band. Snort, Suricata, Zeek. Signature + anomaly.
IPSDetects AND blocks inline. Real-time prevention. Risk: false positives block legit traffic.
VPNEncrypted tunnel. WireGuard = modern standard. Full vs split tunnel. Site-to-site vs remote access.
Packet SniffingCapture network packets. Wireshark, tcpdump. Any plaintext protocol is readable.
ARP SpoofingSend fake ARP replies. Poison victim+gateway ARP tables. MitM position. Defence: DAI.
DNS SpoofingReturn fake DNS responses. Redirect to phishing server. Defence: DNSSEC, DoH.
MitMIntercept + optionally modify traffic between two parties. bettercap, ettercap.
DoS / DDoSOverwhelm target with traffic. Single vs botnet/amplification. CDN for mitigation.
SYN FloodExhaust TCP connection table with spoofed SYNs. Defence: SYN cookies, rate limiting.
AmplificationSpoof source IP in small query → huge response hits victim. DNS=54x, NTP=556x, Memcached=51200x.
Network SegmentationVLANs isolate traffic. Limit lateral movement. Firewall between segments.
Zero TrustNever trust, always verify. Identity-centric. MFA everywhere. Assume breach.
EDREndpoint Detection & Response. Behavioural detection. Detects fileless malware.
nmapPort scanner. -sV=version, -sC=scripts, -A=aggressive, -p-=all ports, --script vuln=CVE scan.
WiresharkGUI packet analyser. Capture + filter. Decodes 2000+ protocols. Display filters.
MetasploitExploitation framework. 2000+ exploits. msfconsole. msfvenom for payloads.
Snort Rulesalert [proto] [src] [sport] -> [dst] [dport] (msg:"text"; sid:N;)
iptablesLinux firewall. -P = policy, -A = append rule, -j = target (ACCEPT/DROP/LOG).
DMZDemilitarised zone. Internet-facing servers. Two firewalls. Separated from internal.
ResponderWindows network poisoning. Captures NTLMv2 hashes from LLMNR/NBT-NS queries.
fail2banAuto-bans IPs after N failed auth attempts. Works with SSH, Apache, nginx, FTP.
AIDEAdvanced Intrusion Detection Environment. File integrity monitoring. Baseline + check.
DAIDynamic ARP Inspection. Switch validates ARP against DHCP snooping table. Blocks ARP spoofing.
WireGuardModern VPN protocol. 4000 lines of code. ChaCha20 + Poly1305 + X25519. Fastest VPN.
